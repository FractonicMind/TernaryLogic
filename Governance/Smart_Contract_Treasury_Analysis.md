# Smart Contract Treasury

**Research Question:** How does on-chain Smart Contract Treasury enforcement interact with DITL hardware enforcement, and what is the honest minimum viable governance architecture that does not depend on software alone?

---

## 3.1 Smart Contract Treasury Vote Rights Architecture

The Treasury's constitutional distinctiveness derives from what it lacks: vote rights. This is not an omission but the defining feature of an autonomous fiduciary.

### Absence of Vote Rights as Design

[VERIFIED] Governance.md Article II Section 2.3: "The Smart Contract Treasury is an autonomous, non-political fiduciary agent. It is not a voting body." The Treasury executes pre-authorized rules; it does not deliberate.

[VERIFIED] The three-body financial constraint from Governance.md:

- Technical Council proposes Treasury disbursement rules via Type 3 vote (≥75% internal supermajority).
- Stewardship Custodians approve or veto those rules via Type 3 vote (≥75% supermajority) for constitutional and ethical alignment.
- Treasury executes autonomously once rules are encoded. No single body can access the Treasury unilaterally. No two bodies can collude to access it without the third being structurally bypassed—the Treasury itself has no discretion to collude.

### Genuinely Autonomous On-Chain Treasury Precedents

[HIGH] As of Q2 2026, no on-chain treasury has been deployed with **complete** autonomy—no admin key, no multisig override, no emergency pause—and survived adversarial testing at scale. The following represent the closest approximations:

| System | Autonomy Level | Override Mechanism | Survival Record |
|---|---|---|---|
| **Uniswap V3 fee switch** | Controlled by UNI governance; fees can be directed by token vote. | Governance can change fee destination. | Not yet activated at protocol level as of 2026; community debate ongoing. |
| **MakerDAO Surplus Buffer** | Autonomous under certain conditions (e.g., flapper auctions). | Governance can adjust parameters. | Survived market stress but governance retains ultimate control. |
| **Yearn Vaults** | Strategies execute autonomously within parameters. | Multisig can emergency pause. | Pause has been used during vulnerabilities (e.g., 2021 exploit). |
| **Gitcoin Grants (Allo Protocol)** | Quadratic funding rounds execute based on community votes. | Council multisig can adjust matching pool allocation. | No major exploits; but centralization persists. |

**Mechanism of limitation**: Every known system retains a God Mode override—a multisig, a DAO vote with low quorum, or a core team admin key. The justification is invariably safety: the ability to pause during an exploit. TL's constitutional prohibition on God Mode (Article VIII, Kill Criterion 1) makes this path unavailable.

[SPECULATIVE] TL's Treasury is [NOVEL] in its combination of:

1. **No admin key**—the Diamond proxy's `diamondCut` function is frozen; the Treasury facet's `upgradeTo` is gated by Joint-Approval.
2. **No pause function**—consistent with No Switch Off.
3. **No discretionary human approval** for individual disbursements.

This means the Treasury must be **correct by construction** at deployment, or its failures are permanent.

### The Milestone Verification Problem

[VERIFIED] Governance.md Article II Section 2.3: The Treasury "shall automatically release funds only upon the successful on-chain verification of pre-defined, automated milestones (e.g., 'protocol upgrade vX.Y is deployed and verified,' 'audit report Z is published to IPFS')."

[HIGH] Who or what verifies the milestone? This is the central vulnerability of autonomous treasuries.

**Verifiable on-chain milestones**: If the milestone is entirely on-chain (e.g., "a specific contract address emits an event with a specific signature"), no oracle is required. The Treasury contract can verify the event directly. Examples:
- A new facet deployment emits an `Upgraded` event; the Treasury verifies the event log.
- A governance vote passes; the Treasury verifies the vote outcome from the Governance facet.

**Milestones requiring off-chain data**: "Audit report Z is published to IPFS" requires an oracle to attest that a file with a specific hash exists on IPFS and that the file meets audit criteria. This introduces a trusted third party. Oracles (e.g., Chainlink, Tellor) are themselves governed by multisigs or token votes and constitute a God Mode vector if they can unilaterally attest to milestones.

[GAP-ARCHITECTURAL] Oracle-free milestone verification is possible only for milestones that are **cryptographically verifiable on-chain**. Examples:
- A zero-knowledge proof that an audit report with specific content has been generated by a known auditor's public key.
- A threshold signature from a set of pre-approved auditors attesting to report publication.

However, this shifts the trust to the auditor set, which must be managed by governance—a recursion that ultimately lands on human attestation.

[CONSTITUTIONAL TENSION] The Treasury's autonomy is bounded by the oracle problem. The Goukassian Principle (No Log = No Action) requires that the Decision Log exist before disbursement. In a software-only system, the oracle that verifies the Decision Log's existence can be compromised or coerced. Only DITL hardware, with its physical commit boundary for audit evidence, can provide oracle-free verification: the Treasury's disbursement transaction would be physically gated until the audit record is written to non-volatile storage and acknowledged by a hysteretic C-element.

### Joint-Approval for Treasury Rules: Paralysis Risk

[VERIFIED] Changing Treasury disbursement rules requires a Type 3 Joint-Approval: 75% supermajority in Technical Council, then 75% supermajority in Stewardship Custodians.

[HIGH] This threshold is identical to major cryptographic upgrades. For routine annual budget adjustments, it may create governance paralysis.

**Precedent**: The U.S. federal budget process requires annual appropriations bills. Failure to pass results in a government shutdown. The high threshold for budget passage in a divided Congress has led to recurring shutdowns and continuing resolutions.[^1] TL's Joint-Approval requirement is even higher (75% of two separate bodies).

[SPECULATIVE] TL could distinguish between **budget parameters** (Type 2, Custodian approval only) and **new disbursement categories** (Type 3, Joint-Approval). Governance.md does not make this distinction; all Treasury rules changes are Type 3. This is a deliberate choice to prevent the Technical Council from unilaterally directing funds, but it may prove operationally brittle.

### Economic Denial-of-Service via Rule Starvation

[HIGH] Attack vector: An adversary with sufficient influence over either the Technical Council or the Custodians prevents Joint-Approval of new Treasury rules. The system continues to collect fees but cannot disburse funds for maintenance, audits, or security upgrades. Over time, the system starves.

**Precedent**: The Ethereum Foundation's funding of client teams is discretionary. No formal on-chain starvation attack has occurred, but the risk of funding capture by hostile actors is recognized in the community.

**Mitigation in TL**:

- The Treasury's existing rules continue to operate. If the rules allocate a fixed percentage of fees to maintenance, those disbursements continue.
- The Technical Council's Type 1 authority (simple majority) allows emergency continuity fallbacks, including Anchor migration. It does **not** allow new Treasury rules.
- If the starvation is sustained, the system degrades but does not halt. This is the "caution as the default posture" principle: inaction is preferable to precipitous action.

[GAP-ARCHITECTURAL] Governance.md provides no mechanism for the Treasury to autonomously adjust disbursement rules in response to prolonged governance paralysis. A minimum viable addition would be a **default fallback allocation**: if no new rules are approved for 24 months, the Treasury reverts to a constitutionally defined minimum allocation (e.g., 30% to node operators, 20% to audit fund).

### Technical Achievability Without Trusted Third Party

[VERIFIED] "Autonomous disbursement on milestone verification" is technically achievable **for on-chain milestones** without a trusted third party. Smart contracts can verify events emitted by other contracts. This is standard DeFi composability.

**For off-chain milestones**, a trusted third party is currently required as of Q2 2026. Zero-knowledge proofs can reduce trust assumptions but cannot eliminate the need for an oracle to attest to off-chain facts. The No God Mode kill criterion makes this an architectural constraint: the Treasury cannot rely on any single entity for milestone verification. A threshold of independent verifiers (e.g., 5 of 7 pre-approved audit firms) could be used, but these verifiers become a new governance surface.

[^1]: U.S. government shutdowns: 2013 (16 days), 2018–2019 (35 days). Source: Congressional Research Service.

[SPECULATIVE] The Joint-Approval mechanism (75% supermajority in both bodies) has no known on-chain implementation that has survived adversarial testing at scale. The closest analog is the Optimism Citizens' House veto, but that body is a multisig in early stages. TL's architecture is [NOVEL] in its combination of two independent on-chain voting bodies with high quorum requirements.

**Gas economics under adversarial conditions**:

[HIGH] An attacker could make governance votes economically infeasible by spamming the network with transactions that inflate gas prices. TL's votes are on-chain transactions requiring signatures from Council/Custodian members. If gas prices spike (e.g., due to an adversary-funded spam attack on the Anchor chains), the cost to cast a vote could become prohibitive.

**Mitigation**: The Treasury could subsidize voting gas costs for Council and Custodian members. This is a permissible Treasury rule under Type 3 Joint-Approval. However, if the attack occurs before such a rule is encoded, members may be unable to vote.

[GAP-ARCHITECTURAL] Governance.md does not specify a gas subsidy mechanism for governance participants. This is an operational detail that could be addressed in the first Treasury rule set.

## 3.3 The Treasury as Autonomous Fiduciary

### Milestone-Gated Autonomous Disbursement: Best Practices

[HIGH] As of Q2 2026, best practices for milestone-gated disbursement in production systems:

- **Gitcoin Grants (Allo Protocol)** : Uses a "round" smart contract where matching funds are distributed based on quadratic voting results. The voting results are computed off-chain and submitted by a trusted operator. This is not fully autonomous.
- **Optimism RetroPGF**: Citizens' House members vote on attestation results submitted by badge-holders. Attestations are off-chain; disbursement is executed by a multisig. Not autonomous.
- **Kleros Court**: Dispute resolution with juror votes; funds are released based on on-chain vote outcomes. This is autonomous for on-chain disputes but relies on human jurors for subjective decisions.

[GAP-ARCHITECTURAL] No production system achieves fully autonomous, oracle-free milestone verification for off-chain milestones. TL's requirement for "autonomous disbursement on milestone verification" is therefore [NOVEL] and currently unproven.

### Proof of Milestone Without Trusted Human Attestation

[HIGH] Cryptographic approaches under research as of Q2 2026:

- **Zero-Knowledge Proofs of computation**: A ZK-SNARK can prove that a specific computation was performed (e.g., an audit tool ran against a codebase) without revealing the inputs. However, the statement "audit report Z is published to IPFS" requires a trusted oracle to attest to the IPFS content.
- **Trusted Execution Environments (TEEs)** : Intel SGX or AMD SEV can attest that code ran in a secure enclave. An enclave could verify IPFS content and sign an attestation. This reduces trust to the TEE manufacturer and is subject to side-channel attacks.
- **Threshold signatures from pre-approved auditors**: A set of independent audit firms sign a statement that report Z exists. The Treasury verifies that ≥66% of the pre-approved set have signed. This is a social trust model encoded on-chain.

[SPECULATIVE] TL could implement the threshold signature approach: the Custodians approve a set of audit firms (Type 2 vote). The Treasury requires signatures from a supermajority of those firms to verify an audit milestone. This does not eliminate trust but distributes it and makes it auditable.

### Economic Attack: Fee Inflation Denial-of-Service

[HIGH] Attack vector: An adversary with substantial capital inflates transaction fees on all Anchor chains simultaneously, making it economically infeasible for TL node operators and governance participants to submit transactions. The system is not halted—it is economically suffocated.

**Precedent**: During the 2021 NFT boom, Ethereum gas prices exceeded 500 gwei for extended periods, pricing out many users. This was market-driven, not adversarial, but demonstrates the effect.

**Mitigations in TL**:

- The Anchor network's multi-chain deployment reduces the impact: an attack on one chain leaves others operational.
- The Treasury could maintain a gas reserve to subsidize critical governance transactions during high-fee periods.
- DITL hardware at the execution layer does not directly address this; it is an economic-layer attack.

### Treasury Autonomy: Feature or Fragility?

[HIGH] The Treasury's autonomy is a **security feature** against capture but a **fragility** against incorrect rules. If the encoded rules produce clearly wrong outcomes—e.g., disbursing funds to a deprecated address, or failing to adjust for inflation—there is no emergency override. The only remedy is a Type 3 Joint-Approval to change the rules, which may take weeks or months.

**Precedent**: The DAO hack (2016) involved a smart contract that autonomously executed rules, resulting in the loss of 3.6M ETH. The community hard-forked to reverse the outcome—a social-layer override of on-chain immutability. TL's constitutional design prohibits such an override, and the Anchor network makes a coordinated hard fork across five chains practically impossible.

[SPECULATIVE] The Treasury must be deployed with **extremely conservative initial rules** and a **slow parameter adjustment mechanism** (e.g., changes take effect only after a 30-day timelock). This allows time for the Custodians to detect and potentially veto a rule change before it executes.

### Three-Party Autonomous Fiduciary Precedent

[GAP-RESEARCH] I do not know of a verifiable precedent where a genuinely three-party autonomous fiduciary model (proposer, vetoer, autonomous executor) has been deployed and survived adversarial conditions. The TL Treasury is [NOVEL] in its constitutional architecture.

## 3.4 The Revocation Contract and Automated Slashing

[VERIFIED] Governance.md Article IX Section 9.5: The Revocation Contract "autonomously applies the pre-approved penalty based on severity... Immediate Access Revocation. The node's credentials are struck from the active-duty registry." An off-chain appeal to the Technical Council or Custodians is available, but "the automated revocation stands unless and until it is overturned by a formal governance vote."

### Current State of Automated Slashing

[HIGH] Automated slashing in proof-of-stake networks as of Q2 2026:

| Network | Slashing Mechanism | Failure Modes |
|---|---|---|
| **Ethereum** | Automated slashing for double-signing and surround voting. Validators are penalized automatically based on on-chain evidence. | False positives are rare but have occurred due to client bugs (e.g., Prysm 2021). Slashed validators have no appeal; funds are burned. |
| **Cosmos Hub** | Automated slashing for downtime and double-signing. | Governance can vote to exempt a validator from slashing in exceptional circumstances (e.g., software bug). This social-layer override has been used. |
| **Polkadot** | Automated slashing for equivocation and unresponsiveness. | Slashed validators can submit a governance proposal for refund; this has occurred for slashings caused by runtime upgrades. |

### TL's Appeal Mechanism Adequacy

[VERIFIED] TL's appeal mechanism is **off-chain** and requires a formal governance vote to overturn revocation. This is a higher bar than Cosmos or Polkadot, where a single governance proposal can refund slashed funds. TL requires a Type 2 vote (≥66% in the Technical Council or Custodians, depending on the nature of the appeal).

[HIGH] Against sensor manipulation or oracle compromise:

- If the Revocation Contract relies on an oracle to detect node misbehavior (e.g., downtime reports from other nodes), a compromised oracle could trigger mass revocation of honest nodes.
- The appeal process is **reactive and slow**. Honest node operators would be removed from the active registry until a governance vote can be held, which may take days or weeks.
- During that period, the network's node count could drop below operational thresholds.

[SPECULATIVE] This creates a **denial-of-service vector**: an attacker could compromise the oracle (or a sufficient number of reporting nodes) to trigger revocation of a large fraction of honest nodes, degrading network performance or liveness. The "automated revocation stands" rule amplifies this risk.

**Mitigation without God Mode**: The Revocation Contract could include a **challenge period** for revocations: a revocation is queued for 48 hours before taking effect, during which the affected node can submit a bond and request expedited review by the Custodians. This does not create an override but introduces a delay that allows human intervention before the node is removed.

[GAP-ARCHITECTURAL] Governance.md does not specify a challenge period. Adding one would be consistent with the Time-Bound Epistemic Hold principle applied to node revocation.

## 3.5 DITL as the Honest Constitutional Floor

### Current Fabrication Status of DITL

[GAP-ARCHITECTURAL] As of Q2 2026, DITL (Delay-Insensitive Ternary Logic) implementations based on NULL Convention Logic remain **theoretical or confined to FPGA prototyping**. The Atomic Auditability paper presents cycle-accurate timing models and FPGA-based emulation results but does not describe deployed silicon in production financial or governance systems. This is explicitly stated in Section VI of the paper: "For the hybrid FPGA/ASIC target platform, we propose a hierarchical approach: FPGA fabric implements the configurable strategy logic... while a hardened 'Ternary Governance Core' ASIC block implements Escrow state enforcement." The ASIC block is a **proposal**, not a deployed component.

[VERIFIED] Therefore, DITL hardware enforcement is a **roadmap requirement**, not available infrastructure. The constitutional governance architecture must operate in a transitional mode where software-layer guarantees are the only deployed enforcement.

### Escrow State as Hardware Instantiation of Epistemic Hold

[VERIFIED] Atomic Auditability paper Section IV.1: "The Spacer state {0,0} is not merely a coding convenience but a physically distinct condition with specific circuit properties. In static CMOS implementation, both rails at logic 0 corresponds to specific transistor configurations that cannot be confused with valid data states even in presence of noise, crosstalk, or power supply variation."

[VERIFIED] Why this cannot be overridden by software, firmware, or OS:

- The Escrow state is enforced by **hysteretic C-elements** (Section V.3). The transition to Executed requires the assertion of a "set" signal that is gated by the audit_valid token.
- No software instruction can lower the voltage rails to bypass the C-element threshold; the transistor configuration physically blocks signal propagation.
- Root access, kernel compromise, or firmware manipulation cannot alter the silicon-level gating without physically modifying the hardware.

### Goukassian Principle Applied to Governance Execution

[VERIFIED] Atomic Auditability paper Section VII.2: LTL formulation: `□(execute → ◇(enter_escrow ∧ record ∧ exit_escrow))`

[SPECULATIVE] Applied to governance contract execution (e.g., a Treasury disbursement), the principle would require:

1. **enter_escrow**: The disbursement transaction is prepared and held in a physically gated state.
2. **record**: The Decision Log (the Joint-Approval vote outcome, the milestone verification proof) is written to non-volatile audit storage and verified.
3. **exit_escrow**: The disbursement transaction is released and included in a block.

This would require that the Anchor chain validators or sequencers run DITL hardware that enforces the Escrow state at the transaction inclusion boundary. As of Q2 2026, no public blockchain validators run such hardware. Therefore, the Goukassian Principle is **not physically enforced** for governance actions in the transitional period.

### Atomic Auditability as Governance Property

[VERIFIED] Atomic Auditability paper Abstract: "execution and audit evidence share the same physical commit boundary."

[SPECULATIVE] "Ghost Governance" is the governance analog of Ghost Fills: a governance action (e.g., Treasury disbursement, facet upgrade) that executes but lacks a corresponding immutable Decision Log due to a failure between commit and audit persistence.

**Elimination requires DITL hardware**: The commit boundary must be physically gated by the existence of the audit record. Software checkpoints (e.g., writing to disk before sending transaction) are "speed bumps" (Section IV.4), not Escrow states. A crash or power loss between write and send can still produce Ghost Governance.

### Residual Vulnerabilities of DITL Hardware

[VERIFIED] Atomic Auditability paper Section VIII.2:

| Vulnerability | Description | Mitigation |
|---|---|---|
| **C-element hysteresis drift** | Threshold voltage drift due to aging, temperature, radiation | Periodic calibration, redundant voting, adaptive biasing |
| **Dual-rail crosstalk** | Aggressive wire spacing causing correlated noise | Shielding, minimum spacing constraints, differential signaling |
| **Completion detection metastability** | Near-simultaneous input arrival | Multi-stage synchronizers, timeout detection, MTBF analysis |

These vulnerabilities are **physical and statistical**, not eliminable. DITL reduces the probability of Ghost Governance to a quantifiable MTBF, but it does not achieve absolute zero.

### Two-Layer Architecture Conclusion

[VERIFIED] Software governance (Layer 2) sits above the DITL hardware floor (Layer 1). Software governance can be upgraded, improved, and corrected through Type 3 Joint-Approval. The DITL hardware floor cannot be overridden by any software-layer action—it enforces the Immutable Mandates at the circuit level.

[SPECULATIVE] In the transitional period (pre-DITL deployment), the software governance layer must operate with the honest acknowledgment that its guarantees are procedural, not physical. The Treasury's autonomy is enforced by smart contract code that executes on mutable infrastructure. This is the honest state of the architecture as of Q2 2026.

## 3.6 Transitional Governance (Pre-DITL Deployment)

### Provisions for the Transitional Period

[VERIFIED] In the absence of deployed DITL hardware, the No Log = No Action invariant is **bypassable**. A software or hardware failure can create a Ghost Governance event.

[HIGH] Transitional provisions should include:

1. **Redundant audit storage**: Decision Logs must be written to multiple independent storage systems (e.g., IPFS, Arweave, Filecoin) before a governance transaction is submitted. This raises the cost of audit loss but does not eliminate the commit-audit gap.
2. **Time-locked execution**: All Type 3 governance actions should be subject to a mandatory timelock (e.g., 48 hours) between approval and execution. During this period, the Decision Log can be replicated and verified by independent observers. This is a software "speed bump," not a hardware Escrow state, but it provides a window for detection of Ghost Governance attempts.
3. **Social-layer monitoring**: The Custodians should operate an off-chain monitoring service that verifies that every executed governance action has a corresponding Decision Log on at least three Anchor chains. Discrepancies trigger a public alert and a constitutional review.

### Acknowledgment of Reduced Enforceability

[SPECULATIVE] The constitutional governance architecture should **explicitly acknowledge** its reduced enforceability during the transitional period. This acknowledgment could take the form of a preamble to the deployed Governance.md that states:

> "This constitution is instantiated in software. The Immutable Mandates defined herein are procedurally enforced by smart contracts and human governance. They are not physically enforced by DITL hardware as of this deployment. The signatories acknowledge that full constitutional protection requires DITL hardware deployment and commit to pursuing that deployment through the governance processes defined herein."

[GAP-ARCHITECTURAL] Whether this acknowledgment should be **on-chain** (e.g., as a permanent record in the Governance facet) is a matter of constitutional philosophy. On-chain acknowledgment creates an immutable record of the transitional state, which is consistent with TL's auditability ethos. Off-chain acknowledgment is less binding but more easily updated when DITL deploys.

### Countering Upgrade Prevention by Adversaries

[HIGH] An adversary who has established influence over the software-layer governance has an incentive to **prevent DITL upgrade**, because DITL hardware would remove their ability to override software constraints.

**Prevention mechanisms**:

- **Delaying Tactical Council approval**: The adversary could use their influence to prevent the Technical Council from proposing a DITL upgrade.
- **Custodian veto**: The adversary could pressure Custodians to veto a DITL upgrade on grounds of "premature deployment" or "unproven technology."

**Countermeasures without God Mode**:

1. **Constitutional mandate for DITL**: Governance.md could be amended (via Type 3 Joint-Approval during the transitional period) to include a statement that "the deployment of DITL hardware enforcement for the Immutable Mandates is a constitutional objective, and any proposal to indefinitely delay such deployment shall be deemed a violation of the No Switch Off mandate." This creates a constitutional obligation for the Custodians to veto indefinite delay proposals.
2. **Treasury earmark**: The Treasury rules could allocate a fixed percentage of fees to a DITL development fund, making the funding autonomous and not subject to annual Joint-Approval.
3. **Anchor network expansion**: The Technical Council could use its Type 2 authority to add new Anchor chains that natively support DITL-like hardware attestation (e.g., chains with TEE-based validation).

### Escrow vs. Speed Bump in Transitional Governance

[VERIFIED] Atomic Auditability paper Section IV.4 distinguishes Escrow state from a speed bump:

| Property | IEX Speed Bump | DITL Escrow State |
|---|---|---|
| Mechanism | Fiber delay | Hysteretic C-element gating |
| Reversibility | Orders may be cancelled | Orders physically blocked |
| Correctness guarantee | None | By construction |
| Audit integration | External | Intrinsic |

[SPECULATIVE] In transitional governance, software timelocks are **speed bumps**. They add latency but do not guarantee that execution and audit share a commit boundary. An adversary with kernel access can bypass a timelock by submitting the transaction directly. Only DITL hardware can provide a genuine Escrow state.

The honest transitional architecture must acknowledge this limitation and rely on the diversity of the Custodian body and the Anchor network to detect and respond to bypass attempts, rather than preventing them physically.

## 3.7 Zombie Governance: Treasury Operation During Layer 1 Compromise

### Unbounded Autonomous Operation

[VERIFIED] Governance.md Article VIII: The "No Switch Off" mandate prohibits any pause, suspend, or kill function. The Treasury Contract is part of the Diamond architecture and inherits this mandate. It cannot be halted by any governance action.

[HIGH] During a 51% attack, consensus failure, or confirmed network compromise on an Anchor chain, the Treasury contract on that chain may continue to execute transactions **if the attacker includes them**. However, the "any one Anchor survives" guarantee means that the canonical Treasury state is defined by the aggregate of all Anchor chains. An attacker who compromises one chain cannot unilaterally alter the Treasury's state on other chains.

### Circuit-Breaker vs. No Switch Off

[VERIFIED] Governance.md contains **no circuit-breaker condition** for halting Treasury disbursement during network compromise. Constitutional fidelity to No Switch Off requires continuous operation regardless.

[SPECULATIVE] This creates a scenario where an attacker with 51% of a single Anchor chain's hashpower could:

1. Censor legitimate Treasury disbursements on that chain.
2. Include fraudulent disbursement transactions that appear valid to that chain's state.

However, the cross-chain notarization requirement (Article VI) means that the canonical Treasury state is the consensus across all 5 Anchor chains. An attack on one chain does not alter the state on the other four. The "any one Anchor survives" guarantee holds: as long as at least one Anchor chain remains uncompromised, the canonical Treasury state is preserved and can be migrated.

### Distinguishing Correct Operation from Exploitation

[HIGH] Without a God Mode, distinguishing "Treasury operating correctly during adversarial pressure" from "Treasury being exploited during network compromise" requires **external verification**:

- The Custodians monitor the Decision Logs across all Anchor chains. A discrepancy in disbursement records across chains indicates compromise on one chain.
- Node operators can choose to follow the canonical state from the uncompromised chains, ignoring the compromised chain's state.

[SPECULATIVE] This is a **social-layer defense**, not an on-chain circuit-breaker. It relies on the Custodians and node operators to detect and respond to compromise. This is consistent with TL's philosophy: governance coordinates; hardware enforces. In the transitional period, the hardware enforcement is absent, so coordination must fill the gap.

### Maximum Financial Exposure from Zombie Governance

[GAP-RESEARCH] I do not know of a verifiable precedent where an autonomous treasury continued to operate during a sustained Layer 1 compromise, leading to financial loss. The closest analog is the **Ethereum Classic 51% attacks** (2019, 2020), where attackers reorganized the chain and double-spent funds. In those events, any smart contract on ETC could have had its state manipulated. No major autonomous treasury was deployed on ETC at the time.

[HIGH] The maximum financial exposure is bounded by:

- The Treasury's balance on the compromised chain. Funds are not custodied by the Treasury across chains; each Anchor chain holds a portion of the total Treasury. An attack on one chain can only affect the funds on that chain.
- The rate of disbursement. If the Treasury disburses slowly (e.g., monthly), the attacker can only extract funds at that rate.

[SPECULATIVE] TL should implement a **cross-chain Treasury model** where funds are held in a multi-chain vault and disbursements require signatures from a threshold of Anchor chains. This would prevent a single-chain compromise from draining funds. This is technically achievable using existing cross-chain messaging protocols (e.g., Chainlink CCIP, LayerZero) and does not require God Mode.

## 3.8 Smart Contract Treasury Verdict

### Core Honest Tension Restated for Smart Contract Treasury

The Smart Contract Treasury is a software-governance execution layer. Its autonomy is procedurally defined and enforced by smart contracts that execute on mutable infrastructure. Any software can be modified by humans, AI systems, or state-level adversaries with sufficient access to the underlying blockchain. The Treasury's "no vote, no pause, no override" design eliminates the most common capture vectors in DAO treasuries, but it does not provide physical enforcement of disbursement integrity. That enforcement requires DITL hardware, where the Goukassian Principle's "No Log = No Action" is a circuit-level interlock, not a smart contract check. The Treasury provides autonomous, accountable, and coordination-resistant funding for the protocol. DITL provides physical guarantee that disbursements occur only when audit evidence exists. Both are necessary. Neither alone is sufficient.

### Treasury's Enforceable Guarantees Given Current DITL Status

As of Q2 2026, with DITL hardware not deployed, the Treasury's guarantees are:

1. **Procedural autonomy**: No single governance body can unilaterally direct Treasury funds. Any disbursement rule change requires Joint-Approval (75% in both Council and Custodians).
2. **Transparent execution**: All disbursements are recorded on-chain and auditable via Decision Logs.
3. **Cross-chain persistence**: The Treasury state is replicated across at least five Anchor chains, making single-chain failure or compromise survivable.
4. **No emergency pause**: The Treasury cannot be halted by any governance action, consistent with No Switch Off.

The Treasury **does not guarantee**:

- Physical prevention of Ghost Governance (execution without audit evidence).
- Oracle-free verification of off-chain milestones.
- Immunity to social-layer capture of the Custodian body.

### Constitutional Integrity Assessment

**[NO CONSTITUTIONAL VIOLATION]** – The Smart Contract Treasury's defined scope remains within the janitor role.

The constitutional test: Does the Treasury's authority change what TL is, or how TL operates within its existing nature?

The Treasury has **no authority**—it executes pre-authorized rules. It does not propose, veto, interpret, or decide. It is the quintessential janitorial function: maintaining the system's financial operation without any discretion to alter the constitutional order. The three-body financial constraint ensures that even if the Technical Council and Custodians collude, they cannot direct the Treasury outside its encoded rules without a Type 3 Joint-Approval—which itself is subject to the same constitutional constraints.

**Emergent architectural authority**: There is a theoretical risk that control over protocol funding could create **de facto** architectural authority. If the Treasury's autonomous rules allocate 100% of fees to a specific development roadmap, that roadmap becomes the de facto future of the protocol, regardless of constitutional intent. The Custodians' mandate review of Treasury rules is intended to prevent this by ensuring that funding purposes align with the Immutable Mandates.

[SPECULATIVE] A structural remedy to prevent emergent architectural authority without God Mode would be a **constitutional cap on Treasury rule scope**: any Treasury rule that allocates more than 50% of fees to a single purpose or that binds the protocol for more than five years must be accompanied by a Custodian finding that it does not constitute architectural change. This is a procedural safeguard, not a hard constitutional amendment, and could be added via Type 3 Joint-Approval.

### Recommended Updates to Governance.md Arising from This Section

1. **Article V (Contract Architecture)** : Add a requirement that the Treasury Contract be deployed with a **default fallback allocation** that activates if no new rules are approved for 24 months. This prevents governance paralysis from starving the system.

2. **Article IX (Workflows)** : Add a **challenge period** for automated node revocation: revocations are queued for 48 hours before taking effect, during which the affected node can post a bond and request expedited Custodian review.

3. **Article II (Treasury)** : Clarify the milestone verification mechanism. Specify that on-chain milestones are verified directly by the Treasury contract; off-chain milestones require threshold signatures from a pre-approved set of independent verifiers, with the verifier set managed by Type 2 Custodian vote.

4. **Transitional Provisions** : Add a new article or appendix acknowledging the reduced enforceability of the Immutable Mandates prior to DITL hardware deployment, and committing the governance bodies to pursue DITL deployment as a constitutional objective.

5. **Cross-Chain Treasury Model** : Specify that Treasury funds are held in a multi-chain vault requiring threshold signatures from a majority of Anchor chains for disbursement, to limit single-chain compromise exposure.
