## The Core Honest Tension

Any software can be modified — by humans, by automated agents, or by state-level adversaries with sufficient access. Constitutional governance written in smart contract code is only as strong as the hardware beneath it.

The sole path to genuine enforcement of the Immutable Mandates is DITL (Delay-Insensitive Ternary Logic) hardware, where Epistemic Hold is not a policy commitment but a physical voltage state (NULL / half-Vdd) that cannot be overridden by software, root access, kernel compromise, or firmware manipulation.

Software governance provides accountability, auditability, and coordination. Hardware governance provides enforcement. Both are necessary. Neither alone is sufficient.

## Three-Body Equilibrium

The governance architecture creates a **three-body equilibrium** preventing consolidation of power:

| Body | Role | Vote Authority |
|---|---|---|
| **Technical Council** | Protocol stewardship, upgrade proposals | Exclusive proposal rights; Type 1/2/3 voting |
| **Stewardship Custodians** | Mandate enforcement, constitutional veto | Binding veto on all Type 3 proposals |
| **Smart Contract Treasury** | Autonomous fiduciary execution | No vote rights — executes jointly approved rules only |

No single body can propose, approve, and execute a change to the protocol. No two bodies can collude to access the Treasury without the third being structurally bypassed.

### Voting Thresholds

- **Type 1** (>50%, Technical Council only) — routine tuning, emergency continuity fallbacks
- **Type 2** (≥66%, respective body) — node certification, Anchor rotation, member confirmation
- **Type 3** (≥75% + ≥75%, Joint-Approval) — Treasury rules, cryptographic upgrades, core contract upgrades

## Key Terminology

| Term | Definition |
|---|---|
| **Epistemic Hold** | State 0. The canonical name for the governance-layer and hardware-layer uncertainty state. Never "pause," "stall," "freeze," or "hold state." |
| **Escrow** | The hardware execution state mapped to NCL NULL/Spacer ({0,0} dual-rail at half-Vdd). Physical instantiation of Epistemic Hold at the circuit level. |
| **Goukassian Principle** | No execution signal may propagate unless the system has entered, recorded, and exited a physically enforced Escrow state. Formalized in Linear Temporal Logic: G(execute → P(escrow_recorded ∧ auditable)). |
| **DITL** | Delay-Insensitive Ternary Logic. The hardware enforcement layer. Pre-production as of Q2 2026. |
| **Atomic Auditability** | Execution and audit evidence sharing the same physical commit boundary. Executed(T) implies Auditable(T) with temporal overlap. |
| **No Log = No Action** | The causal invariant. Maps directly to the Goukassian Principle at the hardware level. |
| **Ghost Governance** | Governance actions that execute without corresponding immutable audit evidence. The governance analogue of Ghost Fills in financial execution pipelines. Eliminated by construction under DITL. |
| **Immutable Mandates** | The exact phrase for the constitutional floor: No Spy, No Weapon, No Switch Off. |
| **Joint-Approval** | Type 3 decisions requiring ≥75% supermajority independently in both Technical Council and Stewardship Custodians. The highest bar in the system. |
| **Time-Bound Epistemic Hold** | When a proposal cannot reach required majority within the decision window, it defaults to rejection at expiration. Caution is the default posture. |

## Confidence Marker Legend

All analytical conclusions in the research documents carry one of the following markers:

| Marker | Meaning |
|---|---|
| `[VERIFIED]` | On-chain facts, formally specified parameters, or direct quotations from source documents |
| `[HIGH]` | Documented real-world precedents with named citations |
| `[SPECULATIVE]` | Logically derived but unproven claims — minimized, always flagged and justified |
| `[GAP-RESEARCH]` | Verifiable precedent of this type exists but was not located — indicates a potential blind spot |
| `[GAP-ARCHITECTURAL]` | TL architecture intentionally leaves this open or defers it to DITL hardware — not a research failure |
| `[NOVEL]` | Feature unique to TL or DITL hardware with no deployed production precedent — accompanied by first-principles derivation |
| `[CONSTITUTIONAL TENSION]` | A structural tension within the architecture that requires monitoring or remedy |
| `[CONSTITUTIONAL VIOLATION]` | A defined scope crossing where a governance body would exercise architect authority — requires structural remedy |
