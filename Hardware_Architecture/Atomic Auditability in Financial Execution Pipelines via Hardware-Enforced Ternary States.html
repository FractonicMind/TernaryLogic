Executive Summary

A physically enforced third execution state ("Escrow/Null") implemented through DITL and NCL eliminates entire classes of race conditions, audit gaps, and irreversibility errors that binary execution fundamentally cannot address.

Executive Summary

Atomic Auditability is achieved when execution and audit evidence share the same physical commit boundary, making execution impossible unless audit evidence already exists.

Binary Execution Problem

Under binary commit semantics, execution precedes audit, creating a temporal window where system failures can produce executed trades without corresponding audit evidence—known as "Ghost Fills."

Ternary Solution

A physically enforced third state—"Escrow" or "Null"—creates an electrical gating barrier where execution paths are physically blocked until audit evidence is irrevocably recorded.

Key Technical Advances

  • Delay-Insensitive Ternary Logic (DITL): Correct operation independent of wire delays or clock skew
  • NULL Convention Logic (NCL): Escrow state mapped to Spacer token for delay-insensitive correctness
  • Hardware-Enforced Escrow: Electrical gating prevents execution until audit verification completes
  • The Goukassian Promise: Formal LTL constraint ensuring atomic auditability

I. Introduction

I.1 Problem Domain: Nanosecond-Scale Execution Correctness

Modern equity and futures matching engines operate at temporal scales where the speed of light through fiber becomes a meaningful constraint on system architecture. The SIX Swiss Exchange X-stream INET platform achieves average latencies of 14 microseconds for OUCH Trading Interface roundtrips, with 93% of transactions completing within 29 microseconds.

The regulatory response to these dynamics has focused substantially on post-trade transparency and audit requirements. Following the 2010 Flash Crash, the CFTC and SEC implemented comprehensive reporting mandates requiring market participants to maintain complete records of all order and execution activity.

Critical Architectural Tension

Audit systems are logically and physically external to execution pipelines, creating a temporal window where system failures can produce executed trades without corresponding audit evidence.

I.2 Ternary Execution Commit Semantics

State Encoding Physical Interpretation
−1: Refused {1,0} dual-rail (FALSE) Order invalid, rejected by pre-trade risk controls or matching engine validation
0: Escrowed {0,0} dual-rail (NULL/Spacer) Physically prevented from committing by hardware-enforced gating. Execution paths electrically isolated.
+1: Executed {0,1} dual-rail (TRUE) Order matched, audit evidence exists in non-volatile storage, execution irreversible

This ternary model restructures the fundamental timing relationship between execution and verification. Under binary semantics, execution precedes audit; the system commits and then records. Under ternary semantics, audit precedes execution; the system records into Escrow state and only then commits.

III. Failure Modes of Binary Commit Semantics

Latency Arbitrage Amplification

Binary commit semantics structurally advantage participants with faster logging infrastructure. System A with 100 ns commit-audit gap can trade against System B's 10 μs gap.

Source: AMF "Ghost Liquidity" Research

Microsecond-Scale Feedback Loops

The 2010 Flash Crash demonstrated how 27,000 E-Mini contracts traded between 2:32-2:45 PM in a self-reinforcing cycle.

Binary semantics provide no natural damping mechanisms.

Cascading Order Book Instability

Latency spikes at one exchange propagate arbitrage opportunities to others. Binary semantics prevent "prepare but do not commit" signaling.

Ghost Fills: Execution-Audit Mismatches

AMF research identifies "Ghost Liquidity"—trades that execute without corresponding audit trail entries.

IV. Hardware-Enforced Ternary Commit Model

Critical Distinction: Escrow is a State, Not a Delay

Unlike IEX's 350μs Speed Bump which adds latency to a synchronous pipeline, DITL creates an asynchronous state barrier that enforces correctness by construction.

DITL Foundations

NULL Convention Logic achieves correctness regardless of wire delays by explicitly representing data validity in signal encoding.

Four-Phase Protocol

Data → Completion → Spacer (Escrow) → Spacer Completion. No direct data-to-data transitions allowed.

Correctness Guarantees

Wire delay independence, clock skew elimination, execution speed orthogonality.

Property IEX Speed Bump DITL Escrow State
Mechanism 38 miles of fiber coil, ~350μs delay Hysteretic C-element gating, electrical state isolation
Timing Effect Adds latency to synchronous pipeline Creates asynchronous state barrier
Reversibility Orders in delay may be cancelled Physically blocked from execution

V. Device-Level Enforcement

Execution-Path Gating

Signal paths electrically blocked through transmission gate configurations where both NMOS and PMOS devices are disabled.

Monotonic Transitions

Escrow-to-Executed transition is irreversible by any physical mechanism other than intended audit-verified path.

Hysteresis

State transition threshold differs from holding threshold, preventing oscillation or accidental triggering.

Hysteretic C-Element Implementation

// Ternary C-element with three stable states
module ternary_c_element (
  input wire set,           // Set condition (audit_valid)
  input wire reset,         // Reset condition
  input wire audit_valid,   // Audit verification token
  output reg Z_f,           // Escrow/Refused rail
  output reg Z_t            // Executed rail
);

  // Escrow state: {0,0} - default condition
  // Executed state: {0,1} - set && audit_valid
  // Refused state: {1,0} - reset || !audit_valid
  
  always_comb begin
    // Pull-up network for Z_t (Executed)
    if (set && audit_valid && !reset)
      Z_t = 1;
    else if (reset || !audit_valid)
      Z_t = 0;
    else
      Z_t = Z_t;  // Hold state - hysteresis
      
    // Pull-down network for Z_f (Refused)
    if (reset || !audit_valid)
      Z_f = 1;
    else
      Z_f = 0;
  end
endmodule

Dual-Rail Encoding Tax Justification

For a major exchange processing 10 million orders per second with average trade value $50,000, a single Ghost Fill event affecting 1,000 trades represents $50 million of un-audited risk exposure.

The dual-rail area cost, amortized across millions of trades, is negligible compared to this risk. The risk elimination justifies the area increase in high-value financial execution contexts.

VII. Formal Constraint: The Goukassian Promise

System Invariant Definition

"No execution signal may propagate unless the system has entered, recorded, and exited a physically enforced escrow state."

VII.2 Linear Temporal Logic Expression

Predicates:
- execute: execution signal asserted
- enter_escrow: system enters Escrow/NULL state  
- record: audit evidence written to storage
- exit_escrow: system exits Escrow state

LTL Formula:
□(execute → ◇(enter_escrow ∧ 
                record ∧ 
                exit_escrow))

Where:
□ (Globally): Property holds at all times
◇ (Eventually): Subproperty holds at some future time
→ (Implication): If antecedent, then consequent
∧ (Conjunction): Both subproperties hold

Stronger Formulation with Until Operator

□(execute → (¬execute U (enter_escrow ∧ ◇(record ∧ exit_escrow))))

The Until (U) operator requires ¬execute hold continuously until escrow conditions are satisfied.

VII.3 SystemVerilog Assertions

// Sequence: proper escrow protocol completion
sequence escrow_protocol;
  enter_escrow ##[1:$] 
  record ##[1:$] 
  exit_escrow;
endsequence

// Property: execute implies prior protocol
property goukassian_promise;
  @(posedge clk or async_handshake)
  disable iff (reset)
  execute |-> ($past(escrow_protocol, 1));
endproperty

// Assertion: runtime check
assert property (goukassian_promise)
  else $error("Goukassian Promise violated");

// Cover: verification
cover property (escrow_protocol ##1 execute);

VIII. Falsifiability and Limits

Conditions of No Advantage

  • Latency-insensitive markets: Execution latency > 1 ms
  • Low-frequency domains: Holding periods > 1 second
  • Error-tolerant applications: Statistical correctness acceptable

Invalidating Assumptions

  • C-element hysteresis integrity: Threshold voltage drift
  • Dual-rail crosstalk immunity: Aggressive wire spacing
  • Completion detection metastability: Synchronizer failure

Sufficient Binary Scenarios

  • Batch processing systems: Complete input before output
  • Single-clock-domain pipelines: Negligible clock skew
  • Approximate computing: Occasional errors acceptable

Falsifiability Criteria

Our central claim—that binary commit semantics fundamentally cannot eliminate certain failure modes—is falsifiable. A demonstration of a binary system with provably zero commit-audit gap under all failure modes would invalidate our argument.

Such a demonstration would require:

  1. Synchronous execution and audit with single-clock-cycle latency
  2. Failure-proof audit storage with acknowledgment before execution visibility
  3. No speculative or pipelined state that could diverge from audited state

These requirements are incompatible with high-frequency trading performance requirements and practical hardware constraints.
