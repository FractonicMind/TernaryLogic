# No Log = No Action: Non-Bypassable Execution Invariant Specification for Ternary Logic

## I. Problem definition: optional logging failure

Traditional computing and control architectures treat logging as a telemetry concern, decoupled from execution by asynchronous buffering, deferred writes, and best-effort delivery semantics. In such systems, the execution path and the persistence path are independent: an action completes on the main processor pipeline while a log entry is enqueued in a memory buffer, forwarded through a network stack, and eventually written to a storage backend. This architectural separation introduces **non-determinism between execution and persistence**, because the temporal ordering of action completion and log commitment is not enforced by any structural constraint. Actions routinely complete while their corresponding log entries remain in volatile buffers, transit queues, or write-back caches, any of which may be lost to power interruption, process termination, memory pressure eviction, or network partition. The result is that the observability layer provides a probabilistic, best-effort record of system behavior rather than a deterministic, complete, and tamper-evident proof of every externally visible effect.

The failure modes of optional logging are numerous and well-characterized. Log entries may be dropped silently when buffer capacity is exhausted under sustained load, producing gaps in the record that are indistinguishable from periods of legitimate inactivity. Log entries may be reordered when multiple producers write concurrently to shared aggregation points without enforced serialization, destroying the causal ordering that reconstruction requires. Log entries may never be written at all if the logging subsystem crashes, is deliberately suppressed by an adversary with kernel-level or hypervisor-level privilege, or is bypassed by execution paths that were never instrumented. In each of these cases, the system has acted, the external world has been affected, and no corresponding record exists to prove, reconstruct, or audit what occurred.

Post-hoc reconstruction, the standard forensic technique applied when logs are incomplete, is fundamentally unreliable under these conditions. Reconstruction depends on correlating timestamps across distributed components, but clock skew, NTP drift, and deliberate clock manipulation introduce timing inconsistencies that make causal ordering ambiguous. Reconstruction depends on cross-referencing partial records from multiple sources, but partial data loss means that the very entries needed to disambiguate conflicting evidence are the ones most likely to be missing. Under adversarial conditions, an attacker with sufficient privilege can selectively delete, modify, or forge log entries, rendering post-hoc analysis not merely incomplete but actively misleading.

Two classes of execution path are particularly dangerous because they reach physical interfaces without traversing any software logging stack. Emergency shutdown sequences in industrial control systems and safety instrumented systems are designed to bypass normal software layers entirely, driving actuators through dedicated hardware circuits to achieve minimum-latency response to hazardous conditions. While this bypass is justified by safety requirements, it creates an unlogged action window during which the system transitions through states that leave no record. **Direct hardware manipulation paths, including Direct Memory Access and memory-mapped I/O**, present an even more severe threat. DMA-capable peripherals, including network interface controllers, GPUs, storage controllers, and any device connected via Thunderbolt, PCI Express, or USB4, can read and write physical memory without CPU intervention or software mediation. A compromised or malicious DMA-capable device can directly write to actuator control registers mapped into the physical address space, triggering externally observable effects with no software-layer awareness and therefore no logging. Even IOMMU protections, which interpose address translation between DMA devices and physical memory, are insufficient: documented vulnerabilities include early boot gaps where IOMMU translation tables are not yet enabled, deferred DMA attacks exploiting stale mappings after virtual machine destruction, and message-signaled interrupt forgery that bypasses IOMMU enforcement entirely. The conclusion is inescapable: telemetry-grade logging provides observability but not enforcement, and any architecture that permits action without structurally guaranteed prior log commitment cannot provide the evidentiary integrity that Ternary Logic demands.

## II. The invariant: No Log = No Action

This specification defines the No Log = No Action invariant as a structural execution constraint within the Ternary Logic framework, requiring that **no state transition, transaction, API call, or physical actuation may be released unless a corresponding log entry has been fully committed to a local hardware-backed non-volatile accumulator**. The invariant is not a policy, a configuration parameter, or a software check; it is an architectural coupling enforced through the physical topology of the execution path, such that the electrical and logical prerequisites for action release include cryptographic proof of prior log persistence.

The invariant is defined over a transition system TS = (S, Act, T, I, AP, L) where S is the set of system states, Act is the set of actions, T ⊆ S × Act × S is the transition relation, I ⊆ S is the set of initial states, AP is the set of atomic propositions, and L : S → 2^AP is the labeling function. Two atomic propositions are distinguished: log(a), which holds when a log entry for action a has been fully committed to durable storage with integrity verified, and exec(a), which holds when action a has produced an externally observable effect or state mutation.

The primary invariant in Linear Temporal Logic, using the precedence pattern with strict Until operator semantics as formalized by Dwyer, Avrunin, and Corbett, is expressed as follows. For all actions a in the action space: ◇exec(a) → (¬exec(a) U (log(a) ∧ ¬exec(a))). This formula states that if exec(a) ever occurs on any execution trace, then exec(a) must not hold until log(a) holds, and at the moment log(a) first holds, exec(a) must still be false. The Until operator is strict: it requires that log(a) eventually becomes true and that exec(a) remains false at every position strictly before and including the position where log(a) is first satisfied. In past-time LTL, the equivalent formulation is □(exec(a) → O(log(a))), stating that at every point in every trace, if exec(a) holds then log(a) must have held at some strictly earlier position, where O is the Once operator with standard past-time semantics.

In Computation Tree Logic, the invariant is expressed over all paths from all reachable states. The CTL formulation is ¬E[¬log(a) U exec(a)], asserting that there exists no computation path on which exec(a) occurs while log(a) has not yet held. Equivalently, A[¬exec(a) W log(a)] states that on all paths, exec(a) does not occur until log(a) has occurred, where the Weak Until operator W permits the possibility that log(a) never occurs, in which case exec(a) never occurs either, which is the correct fail-closed behavior. The strongest formulation combines universal path quantification with the state predicate: AG(∀a ∈ Act : exec(a) → logged(a)), where logged(a) is a state predicate asserting that a complete, hash-verified, integrity-confirmed log entry for a exists in the committed log store.

Log completeness is defined as the conjunction of four conditions. First, canonical serialization: the log entry is produced through deterministic serialization conforming to a fixed canonical form such as JCS (RFC 8785) or deterministic CBOR, ensuring that semantically identical records always produce identical byte sequences. Second, cryptographic hashing: the canonicalized byte sequence is processed through a collision-resistant hash function, producing a digest that is included in the Merkle accumulator and linked to the preceding entry via hash chain. Third, physical persistence: the entry and its hash are written to a hardware-backed non-volatile storage element, such as MRAM, nvSRAM, or battery-backed SRAM, that retains data across power loss without depending on any software-controlled write-back mechanism. Fourth, read-back confirmation with integrity check: after the write completes, the storage subsystem reads back the written data, recomputes the hash, and confirms that the read-back hash matches the originally computed hash, verifying that the data was physically persisted correctly and was not corrupted during the write process. Only after all four conditions are satisfied does the log entry qualify as committed, and only a committed log entry satisfies the log(a) predicate.

Action is defined as any externally observable effect or state mutation, encompassing network packet transmission, actuator signal assertion, database write commitment, API response delivery, financial transaction settlement, sensor reading publication, and any modification to shared state visible to processes or systems outside the logging boundary. Internal intermediate computations that produce no externally observable effect are not actions under this definition, though they may optionally be logged for diagnostic purposes.

The inductive proof obligations demonstrating that all state transitions preserve the invariant follow the standard framework from Baier and Katoen. Let Inv(s) ≡ ∀a ∈ Act : executing(a, s) → ∃e ∈ Log(s) : (e.action = a) ∧ (e.timestamp < exec_time(a, s)) ∧ verified(e.hash, s). The base case requires that for all initial states s₀ ∈ I, Inv(s₀) holds; this is satisfied because no actions are executing in any initial state. The inductive step requires that for all states s and s' such that Inv(s) holds and T(s, a, s') is a valid transition, Inv(s') holds; this is established by the structural constraint that the transition relation T is physically gated such that no transition producing exec(a) in s' can fire unless log(a) is committed in s before the transition. The strengthening condition requires that Inv(s) implies the safety property AG(∀a : exec(a) → logged(a)), which follows directly from the definition of Inv. These obligations can be discharged through symbolic model checking using BDD-based or SAT-based bounded model checking, with the transition system encoding the hardware gate logic, the logging subsystem state machine, and the actuator release mechanism.

## III. Cryptographic actuator interlock and execution coupling

The No Log = No Action invariant transforms logging from an observability mechanism into a protocol-level gate embedded directly in the execution path, enforcing a strict write-before-release model in which no externally observable effect can occur without prior cryptographic proof of log commitment. This enforcement is achieved through a **cryptographic actuator interlock** that couples the physical or logical release of every action to the presentation and verification of a valid execution capability token derived from the committed log state.

The core mechanism is a Merkle accumulator whose root hash serves as the current execution capability token. The Merkle accumulator is an append-only binary hash tree in which each leaf corresponds to a committed log entry, and internal nodes are computed as the hash of their children's concatenation. The root hash of this accumulator is stored in a hardware-protected register, accessible only to the logging subsystem and readable by the actuator interlock, but not writable by any other component regardless of privilege level. This register resides within a hardware trust boundary, implemented through a secure enclave, a dedicated FPGA fabric region, or an ASIC-embedded register file with hardware-enforced access control. Modification of the Merkle root register requires presentation of a valid new root derived from appending a verified log entry to the accumulator, computed and verified entirely within the logging subsystem's trust boundary. Any attempt to execute an action without presenting a valid capability token that matches the current Merkle root triggers an immediate fault condition, halting the action and initiating a safe harbor transition.

Actuator release requires the simultaneous satisfaction of three independent cryptographic requirements. First, the requesting component must present a valid log hash with demonstrated preimage resistance and collision resistance: the hash must correspond to a canonicalized log entry whose preimage is available for verification, and the hash function must provide at minimum 128-bit collision resistance, ensuring that no computationally bounded adversary can construct a distinct log entry that produces the same hash. Second, the requesting component must present a local commit acknowledgment in the form of a hardware-generated attestation quote: a TPM-generated or enclave-generated signed statement binding the current Merkle root, a monotonic counter value, and a freshness nonce, signed by a key that is resident within the hardware trust boundary and has never been exposed in plaintext outside that boundary. Third, inference containment must be verified: for systems operating within the Ternary Logic decision framework, the log entry must include the complete decision vector, the TL state reached (Act +1, Refuse -1, or Epistemic Hold 0), and the confidence metrics that determined the state, ensuring that the decision process is fully captured before its result is released.

This specification states explicitly that no administrative privilege, software override, hypervisor code, or System Management Mode firmware can bypass execution gating. The actuator interlock operates at a layer below all software privilege levels: the electrical path from the decision logic to the external actuator or output interface physically traverses the interlock hardware, and there is no alternate routing that circumvents it. A compromised kernel cannot bypass the interlock because the kernel has no write access to the Merkle root register. A compromised hypervisor cannot bypass the interlock because the interlock's trust boundary is independent of the hypervisor's memory management. SMM code cannot bypass the interlock because the interlock is implemented in dedicated hardware, such as an FPGA fabric or ASIC gate, that is not addressable from the SMM address space.

The transactional semantics of the interlock enforce atomic commit or full abort. Each action proposal initiates a transaction that proceeds through a deterministic state sequence: Idle → Proposed → Logging → Committed → Executing → Complete. The transition from Logging to Committed occurs only when all four log completeness conditions (canonical serialization, hashing, physical persistence, and read-back confirmation) are satisfied. The transition from Committed to Executing occurs only when the three cryptographic requirements (valid log hash, hardware attestation quote, and inference containment) are simultaneously verified. If any condition fails at any stage, the transaction aborts and the system returns to Idle; partial completion is not possible, and no externally observable effect occurs. This atomic semantics ensures that the system cannot enter a state in which an action has partially executed but its log entry is incomplete, or in which a log entry is committed but the corresponding action fails to complete, leaving an orphaned log record without a corresponding state change.

## IV. Hardware root of trust

The No Log = No Action invariant depends on a hardware root of trust that anchors the entire enforcement chain in tamper-resistant physical components, ensuring that the cryptographic guarantees described in the preceding sections cannot be undermined by software compromise at any privilege level. This specification requires a trust chain extending from immutable boot ROM through measured firmware stages to the execution gate hardware, with each link cryptographically verified before control passes to the next.

The foundation of the trust chain is a Boot ROM Root of Trust for Measurement, implemented in mask ROM or one-time programmable memory that cannot be modified after manufacture. This ROM contains the first measurement code, which hashes the next firmware stage and extends the measurement into Platform Configuration Register 0 of a TPM 2.0 module conforming to ISO/IEC 11889. The extend operation, defined as PCR_new = Hash(PCR_old ∥ measurement), is irreversible: PCR values cannot be rolled back or set to arbitrary values, ensuring that the complete ordered sequence of boot measurements is cryptographically committed. Each subsequent boot stage, from UEFI firmware through bootloader through kernel through the logging subsystem initialization, is measured and extended into designated PCRs before execution, creating a measurement chain that can be remotely verified through TPM attestation quotes. The TPM signs the current PCR values along with a verifier-supplied nonce using an Attestation Key that is certified through the TPM's Endorsement Key, which is burned at manufacture and uniquely identifies the physical TPM device.

Secure enclaves, including Intel SGX enclaves and ARM TrustZone secure world partitions, provide execution isolation for critical logging and verification code. Within an SGX enclave, the logging subsystem's hash computation, Merkle accumulator update, and attestation quote generation execute in hardware-protected memory that is inaccessible to the operating system, the hypervisor, and all other software. The enclave's measurement hash (MRENCLAVE) is included in remote attestation reports, allowing external verifiers to confirm that the exact expected logging code is executing on genuine hardware. ARM TrustZone partitions the entire system-on-chip into Secure World and Normal World, with the TrustZone Address Space Controller enforcing memory access control based on the Non-Secure bit on every bus transaction: the logging subsystem and execution gate operate in Secure World, where Normal World software, regardless of privilege level, cannot access secure memory or registers.

HSM-backed modules, conforming to FIPS 140-3 Level 3 or Level 4, provide tamper-resistant key storage and cryptographic operations for the signing keys used in attestation quotes and log entry signatures. Level 3 modules incorporate **tamper-detection circuitry, including conductive mesh layers, light sensors, voltage monitors, and temperature sensors, that triggers immediate zeroization of all plaintext Critical Security Parameters** upon detection of physical intrusion. Level 4 modules extend this protection with a complete envelope of tamper-reactive material and environmental failure protection that detects voltage and temperature manipulation attacks. Keys generated within the HSM boundary never leave that boundary in plaintext; all signing and verification operations execute inside the module.

Physical Unclonable Functions provide device-unique identity that cannot be cloned, extracted, or transferred. PUFs exploit inherent random manufacturing variations in silicon, such as SRAM power-up states or ring oscillator frequency differences, to generate a digital fingerprint that is unique per device and reconstructed on demand from physical characteristics rather than stored in persistent memory. When the device is powered off, no extractable key material exists; when powered on, the PUF generates the root identity key from the physical structure of the silicon itself. Invasive probing disrupts the physical characteristics, rendering the PUF output invalid and ensuring that physical attack on the device destroys the identity it is attempting to extract. The PUF-derived identity is bound to the device's signing key through a certificate chain, ensuring that attestation quotes and log signatures are attributable to a specific physical device.

Side-channel resistant implementation is required for all cryptographic operations within the trust boundary. All hash computations, signature generations, and key derivations must execute in constant time regardless of input data, eliminating timing side channels. Power analysis resistance is achieved through dual-rail pre-charge logic, which equalizes power consumption per clock cycle by computing on complementary wire pairs, and through masking countermeasures that split sensitive intermediate values into random shares. Dynamic frequency scaling driven by a true random number generator further distorts power traces, increasing the measurement complexity required for differential power analysis.

The insufficiency of software-only enforcement is demonstrated through four attack vectors. First, kernel compromise: a compromised kernel has full access to physical memory and can modify logging code, suppress log writes, or forge entries without detection by any software-layer monitor. Second, hypervisor escape: demonstrated through multiple documented vulnerabilities, a virtual machine that escapes its hypervisor sandbox gains access to host memory and can modify or disable any software-based logging mechanism. Third, SMM subversion: System Management Mode operates at a privilege level above the hypervisor and can modify hypervisor memory, write persistent malware to SPI flash, and conceal all activities from operating system detection; SMM callout vulnerabilities allow OS-level attackers to execute arbitrary code at SMM privilege. Fourth, supply chain substitution: compromised firmware, whether UEFI modules, option ROMs, or device firmware, can subvert the entire software stack from before the operating system loads. Only hardware enforcement, where the electrical path to the actuator physically traverses a verification gate that is independent of the CPU's software execution environment, eliminates these classes of bypass.

## V. The Ternary Logic mapping and Epistemic Hold / Solvency Protocol

The Ternary Logic framework defines a triadic decision space T = {+1, 0, −1} that governs every decision cycle in which the system participates. The state +1 (Act) represents a clear affirmative: the system has determined with sufficient confidence that the proposed action is aligned with operational constraints, risk thresholds, and input validity criteria, and the action should proceed. The state −1 (Refuse) represents a definitive negative: the system has determined that the proposed action violates a hard constraint, exceeds a risk boundary, or fails a validity check, and the action must not proceed. The state 0 (Epistemic Hold), designated Solvency Protocol in financial and economic contexts, represents the critical third state: the system has determined that it cannot determine, because **confidence is insufficient, inputs are conflicting, data is incomplete, or the input distribution falls outside the domain for which the decision model was trained or validated**.

This specification requires that every decision cycle produce a log entry recording which of these three states was reached, along with the complete decision vector that determined the state, the confidence metrics computed during evaluation, the raw input data that was presented to the decision function, and the specific trigger condition that caused the state selection. The log entry must be committed to the hardware-backed accumulator before the decision result is released to any downstream consumer, whether that consumer is an actuator, a transaction engine, an API endpoint, or a human operator interface. The No Log = No Action invariant applies to all three states equally: an Act decision that is not logged cannot release the action; a Refuse decision that is not logged cannot release the refusal notification; an Epistemic Hold that is not logged cannot release the hold signal or initiate escalation procedures.

Epistemic Hold is a mandatory state triggered by any of four conditions: confidence below a calibrated threshold, where the threshold is set per decision domain and validated through statistical calibration; conflicting inputs, where two or more input signals provide contradictory evidence that cannot be resolved within the decision cycle's time budget; incomplete data, where required input fields are missing, corrupted, or stale beyond a defined freshness bound; and out-of-distribution inputs, where the input vector falls outside the convex hull or statistical boundary of the training or validation distribution, as detected by distance metrics, anomaly scores, or distribution shift detectors. When any of these conditions is detected, the system enters Epistemic Hold and execution is immediately suspended. During this suspension, all intermediate artifacts must be logged before any escalation or continuation is permitted. These artifacts include the full uncertainty quantification vector containing per-input confidence scores, ensemble disagreement metrics, and calibrated probability estimates; the raw input preservation consisting of the exact byte-level input data as received, before any preprocessing or normalization; and the uncertainty signals comprising the specific detector outputs that triggered the hold, including threshold values, computed scores, and the margin by which the threshold was exceeded or not met.

Forced continuation prevention is enforced through three mechanisms. First, cryptographic time-lock mechanisms embed a verifiable delay function in the Epistemic Hold exit path, requiring that a minimum wall-clock time must elapse between hold entry and any subsequent state transition. The VDF output, computed through sequential squaring in a group of unknown order following the Wesolowski construction, provides cryptographic proof that the required time has elapsed, and this proof cannot be accelerated by parallel computation. Second, external pressure detection through timing analysis monitors the statistical distribution of decision latencies and flags anomalous acceleration as a potential indicator of coercion or adversarial manipulation; deviations beyond a configured number of standard deviations from the baseline latency distribution trigger an automatic audit event. Third, a complete audit trail of all override attempts is maintained: every request to exit Epistemic Hold, whether automated or human-initiated, is logged with the identity of the requestor, the justification provided, the timestamp, and the current system state, and this log entry must itself be committed before the override can be evaluated.

In financial and economic contexts, the 0 state functions as the Solvency Protocol. When an algorithmic trading system, credit assessment engine, or settlement processor encounters ambiguous market conditions, conflicting signals, or data quality degradation, the system enters the Solvency Protocol state, which halts all pending transactions, preserves current positions without modification, and logs the complete market state snapshot, the pending transaction queue, and the specific conditions that triggered the hold. The Solvency Protocol state persists until the triggering conditions are resolved, either through arrival of additional data that restores confidence above the threshold, through human review and explicit authorized continuation, or through expiration of a safety timeout that triggers an orderly position unwinding procedure.

## VI. Cryptographic non-repudiation of action

Every action committed under the No Log = No Action invariant must satisfy verifiable prior log requirements that establish an unbroken chain of cryptographic proof from system initialization through the current state. This chain begins at the boot measurement sequence, where the TPM's PCR values record the identity of every firmware and software component loaded, and extends through every logged decision and action to the present moment. The chain is constructed through the Merkle accumulator, whose append-only structure ensures that each new entry incorporates the hash of all preceding entries into its ancestry path. For any action a executed at time t, a Merkle inclusion proof of O(log n) hash values demonstrates that the log entry for a is a leaf in the accumulator whose root was the active capability token at time t. A Merkle consistency proof of O(log n) hash values demonstrates that the accumulator state at any earlier time t' is a prefix of the accumulator state at t, confirming that no entries were deleted, modified, or reordered between t' and t.

Digital signature binding of actor identity to action commitments ensures non-repudiation at the individual level. Each log entry is signed by the actor's private key, which is held within a TPM, HSM, or secure enclave and has never been exposed in plaintext outside the hardware trust boundary. The signature covers the canonical serialization of the log entry, including the action description, the TL state reached, the decision vector, the timestamp, the monotonic counter value, and the hash of the preceding log entry. The signed log entry constitutes a **cryptographic commitment that the identified actor authorized the specific action at the specific time in the specific system state**, and this commitment cannot be repudiated because only the actor's hardware-held private key could have produced the signature, the key is bound to a PUF-derived device identity, and the monotonic counter prevents replay of signatures from earlier states.

The identity-integrity-logging triad defines the three inseparable components of every auditable action. Identity establishes who authorized the action, through the digital signature verified against a certificate chain rooted in the device's PUF-derived identity and the actor's credential. Integrity establishes that the log entry has not been modified since commitment, through the Merkle inclusion proof and hash chain verification. Logging establishes that the action was recorded before it occurred, through the temporal ordering enforced by the cryptographic actuator interlock. No component of this triad can be satisfied without the other two: an unsigned log entry lacks identity attribution; a signed entry outside the Merkle accumulator lacks integrity verification; and an executed action without a prior committed log entry violates the fundamental invariant.

Invalid execution states are defined and their responses specified. An orphaned action is an externally observable effect for which no corresponding committed log entry can be located in the accumulator; detection of an orphaned action triggers immediate safe harbor transition, key rotation for all signing keys that may have been compromised, and quarantine of the log segment spanning the time window in which the orphan was detected. A hash mismatch occurs when a log entry's stored hash does not match the hash recomputed from the entry's canonical serialization; this triggers safe harbor transition, isolation of the affected log segment, and initiation of a forensic investigation to determine whether the mismatch resulted from storage corruption or adversarial tampering. A signature verification failure occurs when a log entry's digital signature cannot be verified against the expected public key; this triggers safe harbor transition, revocation of the associated signing key, and notification to all relying parties that log entries signed by that key after the last verified entry require independent corroboration.

The aggregate effect of these mechanisms is that the committed log serves as behavioral proof enabling complete trace reconstruction with third-party verification. Any authorized third party, given access to the log, the Merkle accumulator root history, and the public keys of the signers, can independently reconstruct the complete sequence of decisions and actions, verify that every action was preceded by a committed log entry, confirm the identity of every actor, and validate the integrity of every entry, without trusting the system that generated the log or the operator that maintains it.

## VII. Cryptographic primitives and adversarial resistance

Canonicalization is the first defense against ambiguity-based attacks on log integrity. Every log entry must be serialized through a deterministic canonical form before hashing, ensuring that semantically identical records always produce identical byte sequences and therefore identical hash values. This specification requires conformance to RFC 8785 (JSON Canonicalization Scheme) for text-format logs or deterministic CBOR (dCBOR) encoding for binary-format logs. The canonical form defines the exact ordering of fields (lexicographic by key for JSON, deterministic length-first for CBOR), the exact representation of numeric values (IEEE 754 double-precision for JSON), the exact encoding of strings (UTF-8, no BOM), and the prohibition of duplicate keys, optional whitespace, or alternative representations. Without canonicalization, an adversary could construct log entries that are semantically identical to legitimate entries but produce different hashes, enabling substitution attacks that would not be detected by hash chain verification.

Append-only structures form the backbone of tamper-evident log storage. The primary structure is a Merkle accumulator implemented as a binary hash tree conforming to RFC 9162 (Certificate Transparency v2), with each leaf containing the hash of a canonical log entry and each internal node computed as Hash(left_child ∥ right_child). The accumulator supports two fundamental proofs: inclusion proofs demonstrating that a specific entry exists as a leaf in the tree, requiring O(log n) sibling hashes; and consistency proofs demonstrating that the tree at an earlier size is a strict prefix of the tree at the current size, requiring O(log n) hashes and confirming that no entries were deleted, modified, or inserted before existing entries. The hash chain linking successive entries provides a secondary ordering guarantee: each entry includes the hash of its immediate predecessor, creating a linear chain that enforces total ordering independent of the tree structure.

Replay protection is enforced through two complementary mechanisms. Nonce inclusion requires that every log entry contain a cryptographically random nonce generated by a true random number generator within the hardware trust boundary; since each nonce is unique with overwhelming probability, replaying an earlier log entry produces a duplicate nonce that is detected by the accumulator's uniqueness constraint. Monotonic counters, implemented in TPM non-volatile memory or dedicated hardware counter registers, provide an additional ordering guarantee: each log entry includes the current counter value, the counter is incremented atomically with each commitment, and the counter can never be decremented or reset. An attempted replay of an entry with a counter value lower than the current hardware counter value is rejected immediately. Virtual monotonic counters, constructed by maintaining a Merkle tree of counter values with the physical counter protecting the tree root, extend this mechanism to support arbitrarily many independent counter domains without exhausting the limited write endurance of hardware counter storage.

Logging suppression countermeasures address the threat of an adversary who compromises a subset of logging nodes and attempts to silently prevent log entries from being committed. Redundant logging channels with Byzantine fault tolerance ensure that as long as fewer than one-third of logging nodes are compromised, the honest majority can detect and exclude the compromised nodes. This specification requires a minimum of 3f + 1 logging nodes to tolerate f Byzantine faults, following the PBFT protocol of Castro and Liskov, with the Logres protocol of Wanner, Chuat, and Perrig providing formally verified (in Isabelle/HOL) guarantees of agreement (all valid logs produced during a protocol run are equal), completeness (every entry submitted by a client to a correct node is included), and liveness (the protocol always produces a new valid log). Cross-verification between independent log instances, following the Certificate Transparency model of monitors and auditors, detects split-world attacks in which a compromised log presents different views to different verifiers.

Insider threat mitigation requires multi-party authorization and separation of duties for all operations that could affect log integrity. No single individual or credential can modify log retention policies, access raw log storage, rotate signing keys, or approve manual overrides. Critical operations require M-of-N authorization, where N key shares or credentials are distributed across organizationally independent roles and at least M must concur. The separation of duties is enforced technically, not merely procedurally: the logging subsystem, the key management subsystem, the audit verification subsystem, and the override authorization subsystem operate in distinct trust domains with hardware-enforced isolation, and no single compromise can grant access to more than one domain.

Silent action prevention leverages observable energy consumption correlation with logged activity as a physical-layer verification mechanism. In CMOS circuits, instantaneous power consumption is proportional to switching activity; every computational operation produces a characteristic power signature. A monitoring system that captures continuous power traces from the protected system can correlate observed energy consumption patterns against the timeline of logged activities. **Power consumption during periods logged as idle that exceeds the baseline profile indicates unlogged computation, and known operation types produce characteristic signatures that can be matched against a reference library.** While this technique is most effective on dedicated hardware such as FPGAs and embedded controllers, where the power rail is not shared with unrelated processes, it provides a physical-layer tamper detection capability that is independent of and complementary to all software-layer mechanisms.

Post-commit immutability is enforced through write-once storage mechanisms. At the hardware level, WORM (Write Once Read Many) storage ensures that committed log segments cannot be overwritten. Software-layer enforcement through systems such as SealFS, a Linux kernel module implementing transparent tamper-evident logging with forward integrity guarantees, ensures that even if an adversary gains root-level access to the logging host, modifications to committed log data are cryptographically detectable through HMAC ratchet chains. The combination of hardware write-once enforcement and cryptographic tamper evidence ensures that the committed log is immutable in both the physical and logical senses.

## VIII. Failure modes and cyber-physical and financial safe harbor states

This specification requires fail-closed behavior across all critical failure modes, meaning that any failure in the logging, hashing, storage, key management, or communication subsystems results in immediate cessation of action execution rather than degraded or unlogged operation. Storage failure, detected through write acknowledgment timeout or read-back verification mismatch, triggers immediate execution halt: no new actions are released, and the system transitions to safe harbor. Hashing inconsistency, detected when a recomputed hash does not match the stored hash for any entry in the active log segment, triggers execution halt, log segment quarantine, and safe harbor transition. Buffer exhaustion, occurring when the log entry queue reaches capacity and cannot accept new entries, triggers execution halt through backpressure: since no new log entries can be committed, no new actions can be released, and the system effectively self-regulates through the structural coupling of logging capacity to execution throughput. HSM communication loss, detected through heartbeat timeout, triggers execution halt because attestation quotes cannot be generated and therefore the cryptographic actuator interlock cannot be satisfied. Key rotation failure, occurring when the scheduled rotation of signing keys cannot complete due to HSM unavailability, network partition, or M-of-N quorum failure, triggers a grace period bounded by the remaining validity of the current key, after which execution halts if rotation has not succeeded. Certificate expiration triggers immediate execution halt for any action that would require signature verification against the expired certificate, with pre-emptive warning thresholds configured to allow rotation to complete before expiration under normal conditions.

For cyber-physical systems, safe harbor states are defined through three interlocking requirements. The first is **kinetic energy minimization with bounded convergence time**: the system must reduce all kinetic energy (linear and rotational) to zero or to a level below a defined safety threshold within a convergence time T_safe that is bounded and known a priori. The convergence bound is established through Lyapunov stability analysis, demonstrating that a Control Lyapunov Function V(x) exists such that V_dot(x) ≤ −αV(x) for some α > 0 along all trajectories within the safe harbor controller's domain, yielding exponential convergence with time constant 1/α. For finite-time convergence guarantees, the settling-time function T(x₀) provides an upper bound on the time required to reach the equilibrium from initial state x₀. The second requirement is controlled deceleration with bounded jerk and acceleration profiles: the deceleration trajectory must follow an S-curve profile with jerk (the time derivative of acceleration) bounded by a maximum value j_max, ensuring that mechanical structures are not subjected to impulsive loads, vibrations are minimized, and the trajectory is smooth in the third derivative. The seven-segment S-curve profile, consisting of jerk ramp-up, constant acceleration, jerk ramp-down, constant velocity, jerk ramp-up for deceleration, constant deceleration, and jerk ramp-down to zero, provides the reference trajectory for safe harbor deceleration. The third requirement is stable equilibrium: the final state of the safe harbor transition must be a stable equilibrium point of the system dynamics, verified through Lyapunov stability criteria, such that small perturbations do not cause the system to depart from the safe state.

For financial and data-driven implementations, the safe harbor state is defined as immediate transaction halting and liquidity preservation. Upon detection of any logging failure, all pending transactions are frozen in their current state: no new transactions are initiated, no pending settlements are completed, and no position modifications are permitted. Liquidity is preserved by halting all outbound transfers and maintaining current reserve levels. This behavior mirrors the circuit breaker mechanisms employed by securities exchanges, where trading halts at defined thresholds (7%, 13%, and 20% decline from prior close for market-wide circuit breakers) allow the system and its participants to assess conditions before resuming activity. The financial safe harbor persists until logging capability is restored, verified through a full integrity check of the log chain, and explicitly released by authorized personnel through the multi-party authorization procedure.

This specification requires clear separation between high-level decision authority denial and low-level control stability preservation. When the logging subsystem fails and execution is halted, the high-level decision authority (the TL decision engine that selects among Act, Refuse, and Epistemic Hold) is denied the ability to release any new decisions. However, low-level control loops that maintain physical stability, such as motor current regulators, thermal management controllers, and structural load balancers, continue to operate within their pre-defined safe harbor parameters. These low-level controllers do not require high-level decision authority; they execute fixed, pre-validated control laws that maintain the system in its safe equilibrium state. The logging requirement applies to the high-level decision layer; the low-level stability layer operates on pre-committed parameters that were logged at system initialization and do not change during safe harbor operation.

State machines governing failure mode transitions must be model-checked to verify that all transitions lead to safe states. The failure state machine includes states for Normal Operation, Degraded (one or more redundant logging channels failed but quorum maintained), Logging Failure (quorum lost), Safe Harbor Transition (system actively decelerating or halting), Safe Harbor Steady State (system at rest or holding), and Recovery (logging capability restored, integrity verification in progress). The model checker verifies the safety property AG(state ∈ {Normal, Degraded, Safe_Harbor_Transition, Safe_Harbor_Steady, Recovery}), confirming that no reachable state falls outside the defined set, and the liveness property AG(Logging_Failure → AF(Safe_Harbor_Steady)), confirming that every logging failure eventually reaches the safe harbor steady state within the bounded convergence time.

Manual override procedures, required for scenarios in which the safe harbor state must be exited before automated recovery completes, are subject to three constraints. Physical presence is required: at least one authorized operator must be physically co-located with the system, verified through a local authentication mechanism that cannot be satisfied remotely, such as a physical key, biometric sensor integrated into the system's local control panel, or a hardware token inserted into a port on the system chassis. Multi-party authorization is required: override requires concurrent authorization from at least M of N designated authorities, with M ≥ 2 and the authorities drawn from organizationally independent roles. Complete logging is required: all override actions, including the identities of the authorizing parties, the justification provided, the system state at the time of override, and all subsequent actions taken under manual control, must be committed to the log before and during override execution. If the logging subsystem is inoperative (which is the reason the system is in safe harbor), a secondary logging channel, such as a standalone battery-backed recording device co-located with the system, must be employed, and its records must be integrated into the primary log upon restoration.

## IX. Integration with dual-lane architecture

The dual-lane architecture resolves the tension between the sub-millisecond latency requirements of real-time decision systems and the global auditability requirements of distributed verification by separating log commitment into two concurrent but independent lanes, each with distinct latency characteristics, durability guarantees, and failure domains.

The fast lane is defined as local hardware-backed commitment within a **strict sub-two-millisecond latency bound**, implemented through dedicated FPGA or ASIC hardware pipelines that execute the complete log commitment sequence, comprising canonical serialization, SHA-256 hashing, Merkle accumulator update, monotonic counter increment, MRAM or nvSRAM write, and read-back verification, entirely within programmable logic without any dependency on a general-purpose CPU or operating system scheduler. The deterministic pipeline architecture ensures that the worst-case latency is bounded and predictable: each stage of the pipeline has a fixed, known execution time, there are no cache misses, no context switches, no interrupt latencies, and no garbage collection pauses. Battery-backed non-volatile accumulators, implemented through MRAM (which provides native non-volatility through magnetic states with unlimited write endurance) or nvSRAM (which combines SRAM-speed access with automatic SONOS-based non-volatile backup on power loss), ensure that committed log entries survive power interruption without requiring explicit flush operations. The fast lane satisfies the No Log = No Action invariant: the actuator interlock verifies commitment against the fast lane accumulator, and action release occurs only after fast lane commitment is complete. The formal property is AG(exec(a) → fast_committed(a)), stating that on all paths, execution of action a implies that a has been committed in the fast lane accumulator.

The slow lane is defined as asynchronous anchoring to distributed ledgers, timestamp authority services, or multi-party transparency logs for external auditability. The slow lane batches committed fast lane entries into Merkle trees, computes the batch root, and anchors this root to one or more external verifiable data structures. Anchoring mechanisms include RFC 3161 timestamp authority responses, which bind the batch root hash to a trusted timestamp through the TSA's signature; Bitcoin anchoring via the OpenTimestamps protocol, which embeds the batch root in a Bitcoin transaction for global, permissionless verification; and distributed transparency logs conforming to RFC 9162, which provide append-only Merkle tree storage with inclusion proofs, consistency proofs, and public auditability. The slow lane produces Merkle inclusion proofs that link individual fast lane entries to their anchored batch roots, enabling any third-party verifier to confirm that a specific log entry existed at the time of anchoring without accessing the full log.

This specification clarifies that delayed slow lane anchoring does not violate the No Log = No Action invariant. The invariant requires that a log entry be committed to local hardware-backed non-volatile storage before action release; this requirement is satisfied entirely by the fast lane. The slow lane provides additional external auditability, non-repudiation against the system operator, and global temporal ordering, but these are supplementary guarantees beyond the core invariant. An action that has been committed in the fast lane and released through the actuator interlock is valid regardless of whether it has yet been anchored in the slow lane. The formal property is □(exec(a) → fast_committed(a)) ∧ ◇(fast_committed(a) → slow_anchored(a)), stating that execution always requires prior fast lane commitment, and every fast lane commitment is eventually anchored in the slow lane.

Slow lane failures, including network partition preventing communication with timestamp authorities, distributed ledger congestion delaying transaction confirmation, or transparency log unavailability, do not block fast lane operation. The fast lane continues to accept, commit, and release actions at full throughput while slow lane entries accumulate in a local anchoring queue. This isolation is critical: the fast lane's sub-two-millisecond latency guarantee cannot be held hostage to the variable and potentially high latency of distributed consensus protocols. The two lanes operate in separate failure domains with no shared dependencies; a crash, partition, or overload condition in the slow lane has no effect on fast lane commitment or actuator release.

Backpressure and circuit breaker behaviors govern the interaction between the two lanes. When the slow lane anchoring queue depth exceeds a high-water threshold, backpressure propagates to a monitoring dashboard and triggers an alert, but does not affect fast lane throughput. When the queue depth exceeds a critical threshold, the circuit breaker transitions to the open state, suspending new anchoring attempts and allowing the slow lane to recover. During the circuit breaker open state, a half-open probe periodically tests connectivity to the anchoring targets; upon success, the circuit breaker transitions to half-open, allows a limited burst of anchoring requests, and upon sustained success transitions to closed, resuming normal operation. The exponential backoff strategy reduces anchoring attempt frequency during outages, preventing resource exhaustion on both the local system and the anchoring targets.

The gap tolerance between fast lane commitment and slow lane anchoring is bounded by safety-critical time constants that vary by application domain. For cyber-physical systems, the maximum permissible gap is determined by the time horizon over which the system must be able to prove its complete action history to an external verifier in the event of an incident; typical values range from minutes for high-autonomy systems to hours for supervised systems. For financial systems, the gap tolerance is bounded by regulatory reporting requirements, settlement finality deadlines, and audit window durations. This specification requires that the gap tolerance be explicitly defined for each deployment, that the system monitor the current gap duration in real time, and that a gap duration approaching the tolerance bound trigger escalating alerts followed by, at the tolerance bound, a graceful throughput reduction that limits new fast lane commitments to the rate at which the slow lane can anchor them, ensuring that the gap never exceeds the defined tolerance. The formal property bounding the gap is AG(fast_committed(a) → A[¬gap_exceeded U slow_anchored(a)]), stating that from any state where a is fast-committed, the gap tolerance is never exceeded before a is slow-anchored, on all paths where anchoring eventually succeeds.

## Conclusion

The No Log = No Action invariant, as specified across these nine sections, constitutes a binding architectural constraint that transforms logging from an optional telemetry service into a structurally enforced prerequisite for every externally observable effect in a Ternary Logic system. The invariant's enforcement rests on three interlocking pillars: formal temporal logic properties that define the required ordering between log commitment and action release, verified through model checking against the system's complete state space; cryptographic coupling through Merkle accumulators, digital signatures, and hardware attestation quotes that make log entries unforgeable, log sequences tamper-evident, and actor identity non-repudiable; and hardware root of trust mechanisms including TPMs, secure enclaves, PUFs, and HSMs that anchor the entire enforcement chain in physical components resistant to software compromise, side-channel extraction, and physical tampering.

The integration of Ternary Logic's triadic decision space into this enforcement framework ensures that the critical third state, Epistemic Hold, receives the same structural protection as Act and Refuse: the system's admission that it does not know is logged, protected, and auditable with exactly the same rigor as its decisions to act or to refuse. The dual-lane architecture resolves the latency-auditability tension without compromising the invariant, providing sub-two-millisecond local commitment for real-time operation and asynchronous distributed anchoring for global verification. The fail-closed design ensures that any failure in any component of the enforcement chain results in safe harbor transition rather than unlogged operation, preserving the invariant even under hardware failure, adversarial compromise, or operational error. The resulting system does not trust its operators, its software, or its firmware to maintain log integrity; it structurally prevents any of them from violating it.
