# Cryptographic Erasure in Ternary Logic - Step 2: NL=NA Integration, Merkle Architecture Integration, and Commanded Destruction Governance

## Preamble

This document constitutes Step 2 of four in the implementation-ready technical specification for Ternary Logic (TL) Cryptographic Erasure architecture. Step 1 established the threat model, key hierarchy (HKDF-SHA3-256 epoch key derivation, PRF-based subject-derived keys via HMAC-SHA3-512, audit read key branch, five-state key lifecycle), and hardware root of trust (TPM 2.0 baseline with Thales Luna 7 HSM and RISC-V Keystone TEE alternatives, Fast Lane Prohibition with timing budget table, SP 800-90B heartbeat entropy, ternary-binary interface). All Step 1 material is referenced as established and is not re-derived here. Architectural conventions for notation, TL states, Fast Lane Prohibition, NL=NA, Epistemic Hold, Hybrid Shield, and the Anchors pillar carry forward unchanged.

## Section 4: NL=NA Integration and Merkle Architecture Integration

### 4.1 DITL gate interaction with epoch key availability

The NL=NA invariant requires that no computational action proceed without a corresponding Decision Log entry committed and cryptographically sealed before the action executes. The DITL (Decision-In-The-Loop) gate enforces this invariant at the hardware interlock level. Because every Decision Log entry must be encrypted under the current Epoch Key Rotation (EKR) epoch key before it can be Merkle-committed, epoch key availability becomes a hard prerequisite for DITL gate signal generation. If the epoch key is unavailable, the log entry cannot be encrypted; if the log entry cannot be encrypted, it cannot be sealed; if it cannot be sealed, NL=NA prohibits the downstream action from executing. This coupling is not incidental but structurally necessary: the encryption of the log entry is what makes it tamper-evident and what binds it to its epoch for later audit, so any bypass of encryption would undermine the evidentiary chain that NL=NA exists to guarantee.

The gate sequence proceeds in strict serial order. First, log content is generated in volatile working memory within the TEE boundary. Second, the current epoch key is retrieved from the TEE-cached key store. During retrieval, the system is in **State 0-Transient** (Hold, derivation in progress, auto-resolves), reflecting the fact that the key material is being accessed but the gate has not yet opened. The maximum permissible latency for this retrieval is **0.5 milliseconds**, which falls within the sub-2ms Fast Lane budget established in Step 1. Retrieval at this latency is possible because the Fast Lane operates exclusively against TEE-cached epoch keys and never reaches through to the Hardware Root of Trust (HRoT) directly, consistent with the Fast Lane Prohibition. Third, encryption executes: the log entry plaintext is encrypted under the retrieved epoch key using the authenticated encryption scheme specified in Step 1, producing ciphertext and an authenticated tag. Fourth, immediately upon encryption completion, the plaintext log content must be zeroized from all hardware-accessible memory locations within the TEE. No plaintext log content may persist in registers, L1/L2 cache lines, or DRAM pages beyond the encryption operation boundary. The TEE's memory isolation guarantees that plaintext was never visible outside the enclave, but internal cleanup is still mandatory to prevent recovery via cold-boot or enclave-exit side channels. Fifth, the Merkle leaf is constructed from the ciphertext and authenticated tag (specified in Section 4.2). Sixth, the gate signal is released and the system transitions to **State +1** (Execute, gate open), permitting the downstream action to proceed.

Fail-closed behavior governs all error conditions in this sequence. If the TEE-cached epoch key cannot be retrieved for any reason, including TEE hardware failure, cache invalidation due to epoch rotation race condition, or HRoT remote attestation mismatch detected during the TEE's periodic attestation refresh cycle, the system enters **State 0-Suspended** (Hold, governance intervention required). State 0-Suspended is an Epistemic Hold: the system cannot determine whether the key is genuinely unavailable or whether an adversary has tampered with the key store, so it refuses to proceed until a governance actor with sufficient authorization resolves the ambiguity. Plaintext logging as a fallback is **prohibited under any operational condition**. This is a hard architectural constraint, not a configurable policy parameter. The rationale is that a single plaintext log entry outside the EKR envelope would create a permanent gap in the cryptographic erasure guarantee, because that entry could never be retroactively erased via epoch key destruction. Permitting plaintext fallback even in degraded-mode operation would transform every TEE failure into a potential data protection breach, which inverts the fail-safe property that NL=NA is designed to provide.

Destruction Event log entries occupy a unique position in this architecture. When commanded destruction or epoch rotation triggers zeroization of key material (detailed in Section 5), the Destruction Event log entry must itself be EKR-encrypted and must be generated and Merkle-committed **before** zeroization executes. The Destruction Event entry is the cryptographic proof that the destruction was governed, authorized, and logged. It must be encrypted under the epoch key that is about to be destroyed (for subject-scope destruction) or under the succeeding epoch key (for epoch-scope rotation). If encrypted under the key being destroyed, the entry must be committed to the Merkle tree and the tree root must be anchored before the key is zeroized, because after zeroization the Destruction Event entry itself would be unreadable to auditors holding only that epoch's audit key. The implementation must therefore sequence the Destruction Event commit as the penultimate step before zeroization, with a hard barrier preventing the zeroization command from executing until the Merkle commit acknowledgment is received. Zeroization that occurs without a preceding, committed Destruction Event is a protocol violation. Auditors examining the anchor chain can detect this violation by observing a gap in the epoch key sequence or a missing Destruction Event leaf in the Merkle batch covering the relevant time window. This detection capability is what makes NL=NA an auditable invariant rather than merely a design aspiration.

### 4.2 Merkle leaf construction for encrypted entries

Each encrypted log entry produces a Merkle leaf computed as:

```
leaf = H(0x00 || ciphertext || authenticated_tag || epoch_number || entry_sequence || subject_pseudonym_hash)
```

where H denotes SHA3-256 as established in the notation conventions, and the **0x00 prefix** serves as a domain separator distinguishing leaf hashes from internal node hashes, which are computed as H(0x01 || left_child || right_child). This domain separation pattern follows the construction specified in RFC 6962 (Laurie, Langley, and Kasper, "Certificate Transparency," IETF, June 2013, Section 2.1) and carried forward without modification in RFC 9162 (Laurie, Messeri, and Stradling, "Certificate Transparency Version 2.0," IETF, December 2021, Section 2.1.1). Both RFCs state explicitly that the differing hash calculations for leaves and nodes provide second-preimage resistance. The underlying construction traces to Crosby and Wallach ("Efficient Data Structures for Tamper-Evident Logging," 18th USENIX Security Symposium, 2009), which established the append-only authenticated log structure that Certificate Transparency adopted.

The second-preimage resistance property is critical in TL's context because an adversary who could construct an internal node masquerading as a valid leaf could forge log entries without detection. Without domain separation, an attacker knowing the values of two child nodes could present their concatenation as leaf data, producing the same hash as the parent node and thereby inserting a fabricated entry into the tree's verification path. The 0x00/0x01 prefix ensures that the hash input for a leaf always begins with a different byte than the hash input for an internal node, making such forgery computationally infeasible without breaking the preimage resistance of SHA3-256 as specified in NIST FIPS 202 (SHA-3 Standard, August 2015). SHA3-256's sponge construction based on Keccak-f[1600] provides 128-bit collision resistance and 256-bit preimage resistance, and unlike the Merkle-Damgard construction underlying SHA-2, the sponge construction is inherently resistant to length-extension attacks, eliminating a class of vulnerabilities relevant to Merkle tree constructions where an adversary might attempt to extend a leaf hash input.

The leaf structure commits to five fields, each serving a distinct verification purpose. The **ciphertext** and **authenticated_tag** together commit the encrypted content, allowing auditors with the appropriate audit_key_n (derived via the audit read key branch established in Step 1) to decrypt and verify the plaintext. The **epoch_number** binds the entry to a specific EKR epoch, enabling auditors to determine which epoch key governs decryption and to verify that the entry was encrypted under the correct key for its temporal position. The **entry_sequence** is a monotonically increasing counter within each batch, preventing reordering attacks where an adversary swaps the temporal position of entries within an epoch. The **subject_pseudonym_hash** enables identity commitment verification without content access: an auditor who holds pseudonymization keys but not the epoch key can verify that a particular pseudonymous subject's entry exists in the tree and occupies the claimed position, without being able to read the entry's content. This property directly supports the two-layer separation described in Section 4.3.

Epoch boundaries that fall within a 30-second batch window require special handling. Because epoch rotation is asynchronous to batch construction (by design, per Section 4.4's jitter requirements), a batch may contain entries encrypted under two different epoch keys. Each leaf carries its own epoch_number field, so individual leaf verification remains unambiguous. However, the batch Merkle root must also commit to an epoch map that documents which epochs are represented in the batch and provides a one-way commitment to each epoch key. This is achieved by including an additional epoch map leaf for each epoch represented in the batch:

```
epoch_map_leaf = H(0x00 || epoch_number || epoch_key_fingerprint)
```

where **epoch_key_fingerprint = H(epoch_key_n)** is a one-way commitment to the epoch key. The fingerprint reveals nothing about the key itself, since recovering epoch_key_n from H(epoch_key_n) requires breaking the preimage resistance of SHA3-256. The epoch map leaf uses the same 0x00 domain separator as data leaves, maintaining the leaf/node distinction. The epoch map leaf is distinguished from data leaves by a type field encoded within the leaf payload (an implementation detail, not a domain separation concern, since the Merkle tree verifies structural integrity regardless of payload semantics).

The epoch map prevents **epoch substitution attacks** in which an adversary swaps entries between epochs while maintaining individually valid leaf hashes. Without the epoch map, an attacker could take a leaf from epoch n, re-encrypt the same plaintext under epoch m's key (assuming key compromise of both epochs), construct a new leaf with epoch_number set to m, and insert it into a batch for epoch m. The epoch map leaf commits the batch to the set of epoch keys that were active during its construction. A verifier checking the batch can confirm that every data leaf's epoch_number corresponds to an epoch documented in the epoch map, and that the epoch_key_fingerprint for each documented epoch matches the fingerprint derivable from the epoch key provided during audit. If an attacker substitutes an entry from a different epoch, the fingerprint commitment will fail to match unless the attacker can also forge the epoch map leaf, which requires producing a SHA3-256 collision.

The HKDF-SHA3-256 key derivation established in Step 1 uses the info parameter to provide domain separation between epoch keys, following RFC 5869 (Krawczyk and Eronen, "HMAC-based Extract-and-Expand Key Derivation Function," IETF, May 2010, Section 3.2). The info parameter binds each derived key to its epoch context, ensuring that keys derived for different epochs from the same input keying material are cryptographically independent. This domain separation in key derivation complements the domain separation in Merkle tree construction: one prevents key reuse across epochs, the other prevents leaf/node confusion across tree levels.

### 4.3 Hybrid Shield operation order

The Hybrid Shield pseudonymization layer and the EKR encryption layer must execute in a fixed, non-negotiable order within the log entry processing pipeline. **Pseudonymization executes first**, replacing real subject identifiers with pseudonyms according to the Hybrid Shield mapping table. **EKR encryption executes second**, encrypting the pseudonymized content under the current epoch key. This ordering produces a two-layer protection structure where each layer is governed by a separate key under separate authorization, and neither layer alone provides full access to both content and identity.

A verifier who holds pseudonymization keys (the Hybrid Shield mapping) but does not hold the epoch key can verify identity commitments without reading trade-secret or operationally sensitive content. This verifier can confirm that a given pseudonym corresponds to a particular data subject (via the mapping), can verify that entries attributed to that pseudonym exist in the Merkle tree (via the subject_pseudonym_hash in each leaf), and can confirm the structural integrity of the subject's entry chain across epochs. However, because the content is encrypted under the epoch key, this verifier cannot read the substance of any log entry. This capability profile matches the needs of a data protection officer conducting a GDPR compliance review: the DPO must be able to verify that specific data subjects' records exist and are properly managed, but need not read the underlying operational content.

Conversely, a verifier who holds the epoch key but does not hold pseudonymization keys can decrypt and read the content of log entries but cannot resolve pseudonyms to real identities. This verifier sees operational content in full but encounters only pseudonymous identifiers, which are computationally unlinkable to natural persons without the Hybrid Shield mapping. This capability profile matches the needs of a trade-secret auditor or operational reviewer who must assess the substance of logged decisions but has no legitimate need to know which individuals were involved.

Full access to both content and identity requires possession of both the epoch key and the pseudonymization mapping, and this dual access must be authorized through separate governance actions tracked independently in the Immutable Ledger. The requirement for dual authorization is not merely a policy overlay but a structural consequence of the operation order: because pseudonymization precedes encryption, the two protection layers are mathematically independent. Reversing the order (encrypting first, then pseudonymizing) would be architecturally unsound because pseudonymization of ciphertext would not produce meaningful pseudonyms, the pseudonymization function would operate on opaque byte strings rather than identifiable subject fields, and the resulting pseudonym would be bound to the ciphertext rather than to the subject's identity.

The legal basis for this two-layer separation rests on GDPR Article 4(5), which defines pseudonymization as "the processing of personal data in such a manner that the personal data can no longer be attributed to a specific data subject without the use of additional information, provided that such additional information is kept separately and is subject to technical and organisational measures to ensure that the personal data are not attributed to an identified or identifiable natural person." The Hybrid Shield mapping constitutes the "additional information" referenced in Article 4(5), and its storage under separate governance authorization satisfies the requirement that this information be "kept separately." The EDPB Guidelines 01/2025 on Pseudonymisation (adopted January 14, 2025) elaborate on this definition by introducing the concept of the "pseudonymisation domain," the context within which pseudonymisation is intended to preclude attribution to specific data subjects. The Guidelines specify a three-step process: identifying which attributes enable attribution, applying a pseudonymising transformation, and keeping the additional information separate and protected. TL's Hybrid Shield implements each of these steps as architectural invariants rather than procedural controls. The Guidelines further require that controllers assess the risk of attribution within the pseudonymisation domain and ascertain that it is "insignificant," which in TL's architecture is enforced cryptographically rather than probabilistically: without the Hybrid Shield mapping key, attribution is computationally infeasible regardless of the adversary's auxiliary knowledge.

The pseudonymization-first ordering also has implications for the Merkle leaf construction in Section 4.2. The subject_pseudonym_hash field in each leaf is a hash of the pseudonym produced by Hybrid Shield, not a hash of the real subject identifier. This means the Merkle tree itself never contains any direct or indirect commitment to real subject identities. An adversary who compromises the entire Merkle tree, including all leaves and the full tree structure, obtains pseudonymous commitments that are unlinkable to natural persons. Compromise of the Merkle tree combined with compromise of the epoch keys yields decrypted content with pseudonymous identifiers. Only the additional compromise of the Hybrid Shield mapping would complete the chain from Merkle tree to real identity. This three-layer compromise requirement (tree, epoch key, pseudonymization mapping) is a defense-in-depth property that emerges from the operation ordering and is not achievable if the ordering is reversed or if the layers are collapsed.

### 4.4 Anchor timing decoupling

The Anchors pillar provides TL's multi-blockchain anchoring architecture, publishing Merkle roots to external blockchains as tamper-evident timestamps. The TL Constitutional Survivability analysis identified a vulnerability in this architecture: EKR epoch-based timing creates predictable anchor scheduling. If epoch boundaries occur at fixed intervals and each epoch boundary triggers an anchor publication, an adversary can predict exactly when TL will submit transactions to the anchor blockchains. This predictability creates two exploitable attack surfaces.

The first attack surface is targeted network disruption. An adversary who knows that TL will submit an anchor transaction at time t can execute a denial-of-service attack timed to degrade the target blockchain's peer-to-peer network precisely during the submission window. Eclipse attacks, as demonstrated by Heilman, Kendler, Zohar, and Goldberg ("Eclipse Attacks on Bitcoin's Peer-to-Peer Network," 24th USENIX Security Symposium, 2015, pp. 129-144), allow an adversary controlling sufficient IP addresses to monopolize a victim node's connections, isolating it from the honest network. If the adversary can predict TL's anchor submission window, the eclipse attack need only be sustained for that narrow interval rather than continuously, dramatically reducing the adversary's resource expenditure.

The second attack surface is fee market manipulation. On blockchains with auction-based fee markets, an adversary who can predict when TL will submit a transaction can artificially inflate fees during that window. Daian, Goldfeder, Kell, Li, Zhao, Bentov, Breidenbach, and Juels ("Flash Boys 2.0: Frontrunning in Decentralized Exchanges, Miner Extractable Value, and Consensus Instability," IEEE Symposium on Security and Privacy, 2020, pp. 910-927) documented that priority gas auctions on Ethereum cause fee spikes as competing bots bid up gas prices for favorable transaction ordering. Torres, Camino, and State ("Frontrunner Jones and the Raiders of the Dark Forest: An Empirical Study of Frontrunning on the Ethereum Blockchain," 30th USENIX Security Symposium, 2021, pp. 1343-1359) further demonstrated that suppression attacks, where adversaries flood the network with transactions to prevent a victim's transaction from being included in a block, can force the victim to escalate fees or wait multiple blocks for inclusion. An adversary targeting TL could combine fee manipulation with suppression attacks during the predicted anchor window, either forcing TL to overpay for anchor transactions or delaying anchor publication until after the epoch's evidential value has degraded. Saad, Njilla, Kamhoua, Kim, Nyang, and Mohaisen ("Mempool Optimization for Defending Against DDoS Attacks in PoW-based Blockchain Systems," IEEE International Conference on Blockchain and Cryptocurrency, 2019) documented that mempool flooding attacks on Bitcoin spiked transaction fees by artificially inflating mempool size, with a correlation coefficient of 0.69 between mempool size and mining fees observed during the 2017 Bitcoin mempool spam campaigns.

To mitigate these timing-based attacks, epoch boundary timing must incorporate a uniformly distributed random offset drawn from the system's TRNG (SP 800-90B compliant, as established in Step 1). The jitter offset is bounded to **plus or minus 20 percent of the nominal epoch duration**. For an epoch duration of 24 hours, this means the actual epoch boundary may occur up to 4.8 hours before or after the nominal boundary time. The jitter is drawn once per epoch from the TRNG and is not derivable from any public information. Epoch boundaries must not be inferable from blockchain anchor timestamps, which requires that anchor publication operate on an **independent schedule not synchronized to epoch boundaries**. The anchor schedule uses its own TRNG-derived jitter with independent parameterization, ensuring that observing a sequence of anchor timestamps reveals neither the epoch boundary schedule nor the next anchor timestamp.

This decoupling means that an epoch boundary and the corresponding anchor publication are temporally independent events. An epoch may rotate at time t, but the Merkle root covering the epoch boundary entries may not be anchored until time t + delta, where delta is determined by the anchor schedule's independent jitter. The maximum permissible delta between epoch rotation and anchor publication must be bounded to ensure timely evidence preservation, but the bound is a governance parameter, not a fixed interval, and the actual timing within the bound is randomized. Kelkar, Zhang, Goldfeder, and Juels ("Order-Fairness for Byzantine Consensus," CRYPTO 2020, Lecture Notes in Computer Science, vol. 12172, pp. 451-480, Springer) demonstrated that transaction ordering in leader-based consensus protocols is inherently manipulable by malicious leaders, reinforcing the need to decouple TL's anchor timing from any externally predictable schedule.

The jitter offset for each epoch boundary is logged in the corresponding Epoch Rotation Event entry, which is itself EKR-encrypted and Merkle-committed per the NL=NA gate sequence in Section 4.1. Similarly, if a Destruction Event triggers an unscheduled anchor publication, the jitter offset applied to that publication is recorded in the Destruction Event entry. This ensures that auditors can reconstruct the actual timing of all epoch boundaries and anchor publications during a retrospective review, while external observers who lack decryption access cannot extract timing patterns from the encrypted log stream. The jitter values are auditable but not predictable, satisfying both the security requirement (unpredictability to adversaries) and the accountability requirement (reconstructability for authorized auditors).

Remote timing attacks on cryptographic operations, as established by Brumley and Boneh ("Remote Timing Attacks Are Practical," 12th USENIX Security Symposium, 2003, pp. 1-14), demonstrate that even microsecond-scale timing variations can leak information about secret operations when an adversary can make repeated measurements. While the anchor timing jitter operates at a much coarser granularity (hours rather than microseconds), the same principle applies: any deterministic relationship between internal state transitions and externally observable events constitutes a timing side channel. The anchor timing decoupling eliminates the deterministic relationship between epoch rotation (an internal state transition) and blockchain transaction submission (an externally observable event), closing this side channel at the architectural level.

## Section 5: Commanded Destruction - GDPR Erasure Governance

### 5.1 Erasure authorization protocol

Commanded destruction is the governance-driven process by which a specific data subject's records are rendered permanently unrecoverable in response to a validated erasure request. It is architecturally distinct from automated EKR rotation, and this distinction is not merely procedural but structural. Automated EKR rotation destroys an entire epoch key, affecting all subjects and all content within that epoch, and follows the scheduled lifecycle defined in Step 1. Commanded destruction targets a single subject's derivation path within the Subject Derivation Table (SDT), leaving the epoch key intact and all other subjects' data accessible. The two paths have different governance workflows (automated schedule versus human-authorized command), different key scopes (epoch-wide versus subject-specific), and different regulatory documentation requirements (routine key lifecycle management versus individual erasure accountability under GDPR Article 17). Conflating them creates audit ambiguity that could undermine both regulatory compliance demonstrations and forensic investigations.

A valid erasure command requires four sequential prerequisites, each of which must be completed and documented before the next can begin. First, the data subject must submit an authenticated request, and the subject's identity must be verified against the Hybrid Shield pseudonym mapping. Identity verification must confirm that the requesting individual corresponds to the pseudonym whose derivation path will be destroyed. This verification is itself a Hybrid Shield operation requiring access to the pseudonymization mapping, and it must be performed by an authorized operator who holds pseudonymization keys but need not hold epoch keys. The verification step must be logged as a Decision Log entry, encrypted and Merkle-committed per the NL=NA gate sequence, before any subsequent authorization steps execute. GDPR Article 12(3) requires that the controller respond to the erasure request "without undue delay and in any event within one month of receipt of the request," with a permissible extension of two additional months for complex cases provided the subject is informed within the initial one-month period. The TL architecture's timing budget for commanded destruction must fit within this regulatory timeline.

Second, custodian multi-signature authorization is required. For subject-scope destruction (affecting a single subject's derivation path), the default threshold is **3-of-5**: five custodians hold authorization shares, and any three must provide valid signatures to authorize the destruction. For epoch-scope destruction (bulk erasure affecting an entire epoch key, which is a more drastic action affecting all subjects in the epoch), the threshold increases to **5-of-7**: seven custodians, five required. The elevated threshold for epoch-scope destruction reflects the higher authorization bar appropriate for bulk erasure, where a single governance error affects many data subjects simultaneously. These thresholds implement a (k,n) threshold scheme following the construction of Shamir ("How to Share a Secret," Communications of the ACM, vol. 22, no. 11, pp. 612-613, November 1979), which provides information-theoretic security: knowledge of fewer than k shares reveals absolutely no information about the authorization credential. The threshold signature itself is produced via the FROST protocol (Komlo and Goldberg, "FROST: Flexible Round-Optimized Schnorr Threshold Signatures," Selected Areas in Cryptography 2020, Lecture Notes in Computer Science, vol. 12804, pp. 34-65, Springer, 2021; standardized in RFC 9591, June 2024), which enables t-of-n threshold Schnorr signature generation without ever reconstructing the full signing key. FROST's two-round protocol ensures that the signing key is never materialized in any single location during the authorization ceremony, eliminating the single-point-of-compromise vulnerability inherent in schemes that reconstruct the key for signing.

Custodian selection must enforce segregation of duties through the following constraints. No single individual may hold multiple shares across the custodian set. No single organizational unit may control a majority of shares (that is, if the threshold is 3-of-5, no organizational unit may hold three or more shares). At least one share must be held by an independent fiduciary who is external to the operating organization, such as an external auditor, a regulatory representative, or a fiduciary appointed by the TL governance council. These constraints prevent insider collusion scenarios where a single department or reporting chain could unilaterally authorize destruction. The thresholds stated here (3-of-5, 5-of-7) are recommended defaults; the TL governance council must formally set thresholds per deployment domain through an explicit analysis of insider collusion risk that accounts for organizational structure, geographic distribution of custodians, and the sensitivity of the data governed by each deployment. The threshold-setting decision must itself be logged as a governance event in the Immutable Ledger.

Third, a Destruction Event log entry must be generated and Merkle-committed before any zeroization executes. The Destruction Event entry records the authenticated subject request identifier, the pseudonym whose derivation path is being destroyed, the custodian authorization signatures (the FROST threshold signature over the destruction command), the epoch numbers affected, a timestamp, and the HRoT attestation of SDT state prior to destruction. This entry is encrypted under the current epoch key per the NL=NA gate sequence. The Destruction Event's Merkle commitment must be confirmed (that is, the Merkle batch containing the Destruction Event leaf must be constructed and its root computed) before the zeroization command is issued to the HRoT. This ordering is enforced by the DITL gate: the zeroization command is the "action" and the Destruction Event is the "log entry" required by NL=NA before the action may proceed.

Fourth, upon successful zeroization (detailed in Section 5.2), a confirmation receipt is generated for the data subject. The receipt contains the Destruction Event's ledger reference (the Merkle leaf index and batch root hash), enabling the subject to independently verify that their erasure was logged and anchored. The receipt does not contain the epoch key, the pseudonymization mapping, or any information that would allow the subject to decrypt other entries. The receipt is transmitted to the subject through the same authenticated channel used for the original request, completing the erasure workflow within the GDPR Article 12(3) one-month timeline.

The entire erasure authorization workflow is subject to NL=NA: every step, including custodian authorization, Destruction Event generation, zeroization execution, and confirmation receipt transmission, must be logged and Merkle-committed before the corresponding action executes. The invariant applies recursively: the log entries documenting the authorization and destruction are themselves governed by the same NL=NA constraint that governs the original operational log entries whose erasure is being authorized.

### 5.2 Subject key destruction execution

The technical execution of subject-scope destruction targets the Subject Derivation Table (SDT), which maps each subject pseudonym to the epoch-key derivation index that enables computation of that subject's per-epoch encryption key via the PRF-based derivation chain established in Step 1. The SDT is stored in HRoT non-volatile (NV) memory, not in software state, not in a database, and not in any storage medium outside the HRoT's physical security boundary. This placement ensures that the SDT is protected by the HRoT's tamper-detection and zeroization capabilities and that access to the SDT requires authenticated HRoT commands rather than filesystem or database operations.

The destruction operation proceeds as follows. The target SDT entry, identified by the subject pseudonym whose erasure has been authorized per Section 5.1, is located in HRoT NV memory. The entry is overwritten with **cryptographically random data** drawn from the HRoT's internal TRNG, not with zeroes. The choice to overwrite with random data rather than a fixed pattern is deliberate and addresses a specific threat identified by Wei, Grupp, Spada, and Swanson ("Reliably Erasing Data from Flash-Based Solid State Drives," 9th USENIX Conference on File and Storage Technologies, USENIX, 2011). Wei et al. demonstrated that at least one storage controller (SandForce SF-1200) performed block-level deduplication that silently optimized away zero-pattern writes, meaning an all-zero overwrite might not actually program new data into the physical storage cells. While TPM NV memory uses different storage technology than consumer SSDs, the principle generalizes: any controller or firmware optimization that detects and compresses uniform patterns could defeat a zero-based overwrite. Random data is incompressible and unique per write, defeating deduplication or compression optimizations. Hasan and Ray ("Data Recovery from 'Scrubbed' NAND Flash Storage: Need for Analog Sanitization," 29th USENIX Security Symposium, 2020, pp. 1399-1408) further demonstrated that even after page-level scrubbing with zeros, NAND flash cells retain analog threshold voltage imprints that can be exploited to recover prior data. While TPM NV storage is not NAND flash in all implementations, the defense-in-depth principle dictates that the overwrite pattern should be chosen to be maximally resistant to any known recovery technique across all plausible storage technologies.

After the overwrite, the HRoT NV index corresponding to the destroyed SDT entry is deallocated using the TPM2_NV_UndefineSpace command (TCG TPM 2.0 Library Specification, Part 3: Commands, Section 31.4). TPM2_NV_UndefineSpace removes the NV index handle, its public metadata (TPMS_NV_PUBLIC), and releases the NV storage space. For FIPS 140-2 and FIPS 140-3 validated TPMs, the specification requires that NV data be erased when an index is deleted, providing a second layer of sanitization beyond the explicit random-data overwrite. The two-step process (overwrite with random data, then deallocate the index) ensures that both the data content and the metadata are destroyed, and that the destruction satisfies the Purge-level sanitization threshold defined in NIST SP 800-88 Revision 1 (Kissel, Regenscheid, Scholl, and Stine, "Guidelines for Media Sanitization," NIST Special Publication 800-88r1, December 2014, Section 2.5) and its successor SP 800-88 Revision 2 (Chandramouli and Hibbard, September 2025, Sections 3.2.1 through 3.2.5). SP 800-88r2 significantly expanded the treatment of cryptographic erase, requiring that no sensitive data was ever stored in plaintext on the sanitized medium since the last sanitization, that the cryptographic algorithm is NIST-approved, and that the key management meets FIPS 140 standards. TL's architecture satisfies all three requirements: SDT entries are always stored within the HRoT security boundary (never in plaintext on external media), the encryption algorithms are SHA3-family as established in Step 1, and key management is performed by a FIPS 140-validated HRoT.

The critical architectural insight is that the epoch key itself is not destroyed in subject-scope destruction. The epoch key remains escrowed and accessible to authorized auditors via the audit read key branch. What is destroyed is the derivation path: the specific SDT entry that maps the subject's pseudonym to the derivation index needed to compute the subject's per-epoch key from the epoch key via PRF(epoch_key_n, derivation_index). Without this mapping, an auditor who holds the epoch key and attempts to decrypt the subject's entries would need to brute-force the derivation index, which is computationally infeasible given the PRF's (HMAC-SHA3-512) output space. After SDT entry destruction, the subject's ciphertext remains structurally intact in the log and remains Merkle-verifiable. An auditor can confirm that an entry exists at a given position in the Merkle tree, can verify the tree's structural integrity, and can confirm the entry's epoch membership via the epoch_map_leaf. The auditor simply cannot decrypt the entry because the derivation path has been permanently severed. All other subjects in the same epoch, and all trade-secret or operational content in the same epoch not associated with the destroyed subject, remain fully accessible through their intact SDT entries. This selective destruction capability is what makes TL's cryptographic erasure GDPR-compliant without requiring the destruction of entire epochs: the subject's right to erasure is honored while the organization's retention obligations for other subjects' records and for trade-secret content are preserved.

Attestation of destruction is the final step. After the SDT entry has been overwritten and the NV index deallocated, the HRoT produces a signed attestation quote confirming that the specified NV index no longer exists and that the previously occupied storage has been sanitized. This attestation is produced using TPM2_NV_Certify (TCG TPM 2.0 Library Specification, Part 3, Section 31.16), which creates a signed TPMS_ATTEST structure including the NV index name, qualifying data (a nonce to prevent replay), and clock/firmware information, signed by the HRoT's attestation key. Because TPM2_NV_Certify attests to the current state of an NV index, and the index has been undefined, the attestation captures the fact that the index no longer exists by returning TPM_RC_HANDLE for any subsequent read attempt. The implementation approach is to execute TPM2_NV_Certify on the target index before destruction (capturing the pre-destruction state as part of the Destruction Event), execute the destruction, and then use command auditing within an audit session to create a signed record that TPM2_NV_UndefineSpace executed successfully on the specified index handle. This signed quote is appended to the Destruction Event log entry before that entry is Merkle-committed. The quote constitutes the cryptographic proof of erasure: it is a hardware-rooted, non-repudiable attestation that the SDT entry was present, was authorized for destruction, and is now absent. The Destruction Event entry, including the attestation quote, is then committed to the Merkle tree and subsequently anchored per the Anchors pillar schedule, creating an immutable record that the destruction was executed in compliance with the governance protocol.

### 5.3 Regulatory compliance documentation

The chain-of-custody documentation for epoch key escrow must simultaneously satisfy two regulatory regimes that impose superficially conflicting requirements: securities regulations that mandate long-term record retention, and data protection regulations that mandate record erasure. TL's cryptographic erasure architecture resolves this apparent conflict by distinguishing between the content (which can be rendered unrecoverable via key or derivation-path destruction) and the proof of governance (which must be retained indefinitely as evidence that the erasure was properly authorized and executed).

For financial services deployments, SEC Rule 17a-4(f) (17 CFR Section 240.17a-4, as amended effective January 3, 2023, Securities Exchange Act Release No. 96034) requires that broker-dealers preserve records in either non-rewriteable, non-erasable (WORM) format or, under the 2022 amendment, using an audit-trail alternative that maintains a complete time-stamped record of all modifications and deletions, the date and time of each action, the identity of the individual performing the action, and sufficient information to permit recreation of the original record and its interim iterations. TL's Merkle-committed, blockchain-anchored log satisfies the audit-trail alternative: every log entry is append-only in the Merkle tree (structural immutability), every modification or access is itself logged via NL=NA (complete audit trail), timestamps are embedded in both the Merkle batch structure and the blockchain anchor (time-stamping), and the FROST threshold signature on governance events identifies the authorizing custodians (individual identification). FINRA Rule 4511 requires preservation for at least six years for records without a specified retention period, with the first two years in an easily accessible location, and requires all records to be preserved in a format compliant with SEC Rule 17a-4. TL's architecture satisfies this by maintaining the encrypted log in perpetuity (the ciphertext is structurally permanent in the Merkle tree and anchor chain) while the epoch keys required for decryption are escrowed with access controls governed by the key lifecycle.

Access grants to escrowed epoch keys are logged to the Immutable Ledger. Every request to retrieve an escrowed epoch key, whether for audit, for regulatory examination, or for litigation hold, generates a Decision Log entry that passes through the full NL=NA gate sequence: the access request is logged, encrypted, and Merkle-committed before the key is released from escrow. This ensures a complete chain of custody for every key access event. Unauthorized access attempts, defined as requests that fail authentication, that originate from unauthorized network segments, or that attempt to bypass the custodian authorization threshold, trigger an immediate transition to **State -1** (Block, zeroization complete, permanent) for the targeted key escrow path and generate a governance alert. The governance alert is itself a mandatory logged event subject to NL=NA: the alert generation and notification are logged and Merkle-committed before the alert is transmitted to governance actors. This prevents an adversary from suppressing the alert by attacking the logging infrastructure, because the alert's Merkle commitment is anchored independently of the alert delivery mechanism.

Retention periods for Destruction Event log entries are governed by the more stringent of applicable regulatory requirements. For financial records subject to SEC Rule 17a-4, the minimum retention period is the statutory period required by the applicable record category: six years for general ledgers and account records, three years for correspondence and communications. For records subject to GDPR accountability obligations, the retention period derives from Article 5(2), which requires the controller to "be able to demonstrate compliance" with data processing principles, and Article 30, which requires maintenance of records of processing activities including "where possible, the envisaged time limits for erasure." In practice, the accountability obligation means that Destruction Event log entries must be retained for as long as the controller may need to demonstrate that an erasure request was properly received, authorized, and executed. Because supervisory authorities may inquire about past erasure actions at any time, and because the limitation period for GDPR enforcement actions varies across Member States (with some having no statute of limitations for certain violations), the practical retention period for Destruction Event entries is **indefinite**.

This indefinite retention does not contradict the erasure that the Destruction Event documents. The Destruction Event log entry is not the personal data; it is the proof that personal data was destroyed. The entry contains the pseudonym (not the real identity), the destruction authorization signatures, the HRoT attestation quote, and the Merkle commitment reference. It does not contain the subject's actual data, which was encrypted under an epoch key whose derivation path for that subject has been permanently severed. The retention of the Destruction Event entry is itself a data-minimization-compliant act: it retains the minimum information necessary to demonstrate compliance, in pseudonymized form, consistent with the accountability principle. The EDPB's coordinated enforcement framework report on the right to erasure (published February 2026 following 2025 coordinated investigations by 32 Data Protection Authorities) recommends that controllers maintain "a documented and updated internal procedure to handle erasure requests" and that the absence of such documentation was identified as a major compliance deficiency. TL's architecture exceeds this recommendation by not merely documenting the procedure but cryptographically committing every instance of the procedure's execution to a tamper-evident, hardware-attested, blockchain-anchored log.

The SEC Rule 17a-4(f) WORM requirement is satisfied by the combination of the Merkle tree's append-only structure and the blockchain anchor's immutability. Once a Destruction Event entry is Merkle-committed and the batch root is anchored, the entry cannot be altered or deleted without invalidating the Merkle root, which would be detectable by any party holding a copy of the anchor transaction. The 2022 amendment's audit-trail alternative is also satisfied, as described above. FINRA Rule 4511(c) requires that all records be preserved in a format compliant with SEC Rule 17a-4, which TL satisfies through the same mechanism. The convergence of these regulatory requirements with TL's architectural properties is not coincidental: the architecture was designed so that the structural properties required for cryptographic erasure (append-only logs, tamper-evident commitments, hardware-rooted attestation) are the same structural properties required for regulatory records retention. The Merkle tree and anchor chain serve dual purposes, providing both the evidence that data existed (for retention compliance) and the evidence that data was properly destroyed (for erasure compliance), without contradiction.
