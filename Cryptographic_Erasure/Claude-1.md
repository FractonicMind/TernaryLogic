# Cryptographic Erasure in Ternary Logic - Step 1: Threat Models, Key Hierarchy, and Hardware Root of Trust

**Ternary Logic Technical Specification - Cryptographic Erasure Architecture**
**Document Scope:** Step 1 of 4
**Version:** 1.0
**Architectural Domain:** Epoch Key Rotation, Hardware Root of Trust, Threat Modeling

## Preamble and architectural scope

Ternary Logic (TL) mandates that every governed action produce a cryptographically sealed Decision Log entry committed to an immutable append-only ledger anchored across multiple blockchains. This design creates an inherent tension: the same immutability guaranteeing accountability becomes an attack surface for trade-secret extraction and an obstacle to GDPR Article 17 erasure obligations. **Cryptographic erasure resolves both tensions through a unified key-destruction primitive** that renders protected content computationally unrecoverable without altering the ledger's Merkle commitment structure. This document, Step 1 of four, specifies the threat models that motivate cryptographic erasure, the key hierarchy architecture that enables it at both epoch and data-subject granularity, and the hardware root of trust that anchors epoch counters, entropy, and zeroization in tamper-responsive silicon. All constructions use binary cryptographic primitives (AES-256-GCM, HKDF-SHA3-256, HMAC-SHA3-512) for hardware-acceleration compatibility and standards compliance. Ternary logic governs execution state transitions between cryptographic operations, not the operations themselves: the three-valued state machine (State +1 Execute, State 0 Hold, State -1 Block) gates key release, pipeline access, and zeroization at the hardware-software boundary.

Throughout this specification, the following notation applies. The symbol || denotes byte-string concatenation. H(x) denotes SHA3-256(x) as defined in NIST FIPS 202 (August 2015). PRF(k,v) denotes HMAC-SHA3-512(k,v), producing a 512-bit pseudorandom output. KDF(ikm, salt, info) denotes the HKDF-SHA3-256 construction following the extract-then-expand paradigm specified in RFC 5869 and analyzed by Krawczyk at CRYPTO 2010 (Springer LNCS vol. 6223, pp. 631-648). The term "zeroization" follows the FIPS 140-2 definition: a method of erasing electronically stored data by altering or deleting the contents of data storage to prevent recovery of the data, distinct from software overwrite in its immediacy and hardware-enforced irreversibility.

## Section 1: Forward secrecy threat model and erasure threat model

### 1.1 Trade secret threat model

The Decision Log justification fields within the TL ledger carry proprietary intelligence that constitutes protectable trade secrets under the Defend Trade Secrets Act (18 U.S.C. 1836) and the EU Trade Secrets Directive (2016/943/EU). These fields include model configuration parameters, risk scoring thresholds, counterparty inference chains, algorithmic weighting coefficients, and governance escalation decision rationales. **Not every field in a Decision Log entry requires confidentiality protection.** The ledger design distinguishes between audit-accessible fields stored in plaintext or under audit-key encryption (timestamps, event type identifiers, governance state transitions, Merkle batch indices, and blockchain anchor transaction hashes) and EKR-encrypted fields (justification narratives, proprietary parameters, model configurations, risk thresholds, counterparty identifiers, and any content from which competitive intelligence could be reconstructed). Audit-accessible fields enable regulatory verification of the governance process without exposing the commercial substance of decisions.

Three adversary classes define the trade-secret threat surface. The first class, the passive archive reader, possesses read access to committed ciphertext on the ledger, including all blockchain anchor layers, but holds no epoch keys and performs no active protocol interference. This adversary attempts offline brute-force decryption, side-channel inference from ciphertext structure, and long-horizon cryptanalytic attacks including future quantum capabilities. Defense against this class requires that AES-256-GCM ciphertext be computationally indistinguishable from random under the chosen-ciphertext security model (IND-CCA2), which GCM achieves when nonces are never reused within an epoch, a property enforced by the deterministic nonce derivation scheme in Section 2.1. AES-256 provides **128-bit post-quantum security** under Grover's algorithm, which halves the effective symmetric key strength. As confirmed by NIST's post-quantum cryptography standardization program and the NSA Commercial National Security Algorithm Suite, this security margin eliminates the need for symmetric cipher migration; post-quantum migration applies exclusively to key establishment (ML-KEM per FIPS 203, August 13, 2024) and digital signatures (ML-DSA per FIPS 204 and SLH-DSA per FIPS 205, both August 13, 2024).

The second adversary class, the active partial-compromise adversary, has obtained the epoch key for some epoch N through insider theft, coerced disclosure, or exploitation of an implementation flaw, but does not possess keys for any other epoch. This adversary can decrypt all EKR-encrypted fields within epoch N. The architectural response is strict epoch-bounded forward secrecy: **compromise of epoch N must not enable recovery of epoch N-1 keys or earlier**, and must not enable derivation of epoch N+1 keys or later. Forward secrecy in the original formulation by Diffie, van Oorschot, and Wiener (Designs, Codes and Cryptography, vol. 2, no. 2, pp. 107-125, 1992) requires that compromise of long-term secrets not reveal previously established session keys. The TL epoch construction adapts this property to the symmetric domain following the taxonomy developed by Boyd, Gellert, Jager, and Stebila (The Computer Journal, vol. 64, no. 4, pp. 639-652, 2021), which identifies "windowed forward secrecy" as the property that keys within each time window (epoch) are cryptographically independent. This independence is achieved by three mechanisms operating jointly: each epoch key derivation incorporates non-recoverable entropy from hardware true random number generation (Section 3.3), the HKDF extract phase acts as a one-way randomness extractor that cannot be inverted to recover input keying material (Krawczyk, CRYPTO 2010), and the previous epoch key undergoes hardware zeroization rather than software overwrite upon rotation completion, ensuring no residual electromagnetic or charge-state artifacts persist in storage media.

The third adversary class, the metadata adversary, holds no keys but analyzes observable metadata: ciphertext lengths, timing of ledger commits, frequency of blockchain anchor transactions, and patterns of epoch rotation events. Ciphertext-length analysis can reveal approximate decision complexity; commit timing can reveal operational tempo; and anchor frequency variations can signal abnormal governance events such as emergency rotations. Mitigation of metadata leakage is achieved through constant-rate Merkle batch commitment (one batch every 30 seconds regardless of decision volume, with padding entries when necessary), fixed-length ciphertext padding of EKR-encrypted fields to the nearest 256-byte boundary, and normalization of blockchain anchor cadence to a fixed schedule independent of batch content density. These countermeasures reduce metadata leakage to the minimum inherent in operating a timestamped append-only ledger on a public blockchain.

### 1.2 Erasure threat model

Article 17(1) of the General Data Protection Regulation (Regulation (EU) 2016/679) establishes the data subject's right to obtain erasure of personal data without undue delay where the data is no longer necessary for the purpose for which it was collected, consent has been withdrawn, the data subject objects to processing, or the data has been unlawfully processed, among other grounds. Article 17(3) specifies exceptions including compliance with legal obligations and the establishment, exercise, or defense of legal claims. The TL ledger's immutability creates a direct conflict: **physical deletion of any entry would break the Merkle commitment chain**, invalidating all subsequent hash linkages and destroying the evidentiary guarantee that is the ledger's primary purpose. Every entry's hash is incorporated into the batch Merkle root, which is itself committed to multiple blockchain anchors. Removing an entry would require recomputing the entire chain from the deletion point forward and re-anchoring every affected batch, an operation that is architecturally prohibited because it would violate the append-only invariant and would be detectable by any party holding a prior anchor commitment.

Cryptographic erasure, also termed crypto-shredding, resolves this conflict by destroying the encryption key that protects subject-specific data, rendering the ciphertext permanently unrecoverable while leaving the Merkle structure intact. This technique was first formalized by Boneh and Lipton in "A Revocable Backup System" (Proceedings of the 6th USENIX Security Symposium, pp. 91-96, July 1996), which demonstrated that data destruction can be separated from physical media access by relying on key management: destroying the encryption key is functionally equivalent to destroying the data itself when the encryption is computationally secure. Reardon, Basin, and Capkun systematized the field in "SoK: Secure Data Deletion" (Proceedings of the 2013 IEEE Symposium on Security and Privacy, pp. 301-315), classifying crypto-shredding as a cryptographic deletion technique operating at the key-management layer rather than the storage layer.

The legal acceptability of cryptographic erasure under GDPR rests on Recital 26, which states that data protection principles do not apply to information rendered anonymous "in such a manner that the data subject is not or no longer identifiable." When the decryption key for a data subject's records is destroyed and no alternative path to the plaintext exists, the remaining ciphertext ceases to constitute personal data under this definition. NIST Special Publication 800-88 Revision 1 (December 2014, with Revision 2 published September 2025) explicitly recognizes Cryptographic Erase as a valid "Purge"-level sanitization technique, defined as "a method of sanitization in which the media encryption key for the encrypted Target Data is sanitized, making recovery of the decrypted Target Data infeasible."

The European Data Protection Board addressed this question directly in Guidelines 02/2025 on Processing of Personal Data through Blockchain Technologies, adopted April 8, 2025 for public consultation. The EDPB acknowledges that "upon deletion of this decryption key, the encrypted data will be unintelligible, at least until the algorithm is broken, the decryption techniques advance sufficiently to allow the decryption of the cipher text, or if the key had already been compromised or leaked." For cryptographic commitments with perfectly hiding properties, the EDPB states that "once the original data and its witness are deleted, the commitment persisting in the blockchain is useless" and that "it will be neither possible to recover nor to recognise the original personal data." However, the EDPB introduces two important caveats: encrypted personal data remains personal data prior to key destruction, and "even state-of-the-art encryption perfectly implemented will be overtaken by time if the blockchain is retained indefinitely," reflecting concern about long-horizon quantum cryptanalytic risk.

The Commission Nationale de l'Informatique et des Libertes (CNIL) addressed blockchain erasure in its September 2018 report "Blockchain et RGPD: quelles solutions pour un usage responsable en presence de donnees personnelles." The CNIL concluded that cryptographic techniques, "without being able to lead to strictly identical effects" as physical deletion, "allow for coming closer to GDPR compliance requirements, notably by cutting off accessibility to the data." The CNIL identified perfectly hiding cryptographic commitment schemes as the strongest technical approach and stated that the equivalence of other techniques "with GDPR requirements must be evaluated" on a case-by-case basis. The UK Information Commissioner's Office has not issued specific guidance on crypto-shredding but maintains a "beyond use" doctrine under which data is considered effectively deleted if the controller is not able and will not attempt to use the data to inform any decision about any individual, does not provide access to any other organization, surrounds the data with appropriate security, and commits to permanent deletion if and when it becomes possible.

The EDPB Coordinated Enforcement Framework report of February 2026 on the right to erasure, covering investigations by 32 DPAs across the EEA examining 764 controllers, found that regulators are actively scrutinizing anonymization techniques used as alternatives to deletion, requiring "sufficient guarantees for irreversible nature." The Agencia Espanola de Proteccion de Datos published a proof-of-concept technical note on blockchain erasure compliance that the EDPB cites in footnotes 8, 24, and 41 of Guidelines 02/2025, indicating regulatory recognition of governance-plus-technology approaches.

**A residual legal risk persists**: some jurisdictions may not accept cryptographic erasure as equivalent to physical deletion, particularly where enforcement authorities interpret "erasure" strictly as physical removal of data. TL deployments must disclose the use of cryptographic erasure to the competent supervisory authority before deployment and must include in their Data Protection Impact Assessment an analysis of encryption algorithm longevity, key management integrity, and the quantum-resilience of the chosen cipher suite. The AES-256-GCM baseline provides 128-bit post-quantum security, substantially exceeding the threshold at which the EDPB's "overtaken by time" concern applies within any foreseeable regulatory horizon.

The scope of commanded destruction in the TL architecture applies to the derived key for a specific data subject, not the epoch key. Destroying an epoch key would render all content within that epoch unrecoverable, including data belonging to subjects who have not requested erasure and audit evidence required for regulatory compliance. Subject-scope destruction targets only the mapping entry in the Subject Derivation Table (specified in Step 2, Section 5.2) that enables re-derivation of the subject's per-epoch encryption key. Once this mapping is destroyed, the subject's pseudonym can no longer be resolved to a derivation input, and the PRF construction (HMAC-SHA3-512) prevents recovery of the subject key without the exact pseudonym-epoch pair. All other subjects' data within the same epoch remains fully recoverable.

### 1.3 Unified destruction guarantee

Both the trade-secret forward-secrecy requirement and the GDPR erasure requirement resolve to the same cryptographic primitive: key zeroization through the hardware root of trust. The distinction lies in scope. Epoch-scope destruction, triggered by epoch key rotation or emergency zeroization, destroys the epoch encryption key itself, rendering all EKR-encrypted content within that epoch permanently unrecoverable. This operation executes on every normal epoch boundary (default 24 hours) and on any forced rotation event (heartbeat entropy failure, tamper detection, governance command). Subject-scope destruction, triggered by a commanded erasure request, destroys only the Subject Derivation Table entry for the requesting data subject, rendering that subject's derived key non-re-derivable while preserving all other subjects' data and the epoch key itself.

Both scopes invoke the same hardware zeroization path: the hrot_zeroize() interface call, which triggers FIPS 140-3 compliant key destruction in the hardware root of trust. For epoch-scope operations, hrot_zeroize() targets the epoch key material in the HRoT's protected storage. For subject-scope operations, hrot_zeroize() targets the Subject Derivation Table entry in TEE-shielded memory. The physical mechanism is identical: voltage-level clearing of the target memory region with hardware verification that no charge-state residue persists, per the zeroization requirements of FIPS 140-3 (approved March 22, 2019; mandatory for new cryptographic module submissions from April 1, 2022), which requires zeroization of all unprotected Sensitive Security Parameters at all security levels.

Both scopes produce the same class of audit evidence. Before zeroization executes, a Destruction Event record is committed to the Immutable Ledger containing the destruction timestamp, the scope identifier (epoch or subject), a cryptographic commitment to the key material being destroyed (a hash of the key, not the key itself), the governance authorization chain that approved destruction, and the HRoT attestation quote proving the counter state at destruction time. This Destruction Event is stored in the audit-accessible plaintext layer of the ledger, ensuring it survives the very zeroization it authorizes. **The TL "No Log = No Action" invariant applies unconditionally**: no zeroization can execute until the Destruction Event has been committed to the current Merkle batch and the batch has been closed. If the system cannot log the Destruction Event (ledger unavailable, Merkle batch failure, blockchain anchor timeout), the zeroization is deferred to State 0-Suspended until the governance pipeline can accept the log entry. This creates a complete, tamper-evident audit trail of every destruction action without retaining the destroyed content.

## Section 2: Key hierarchy architecture

### 2.1 Epoch key derivation via HKDF-SHA3-256

The epoch encryption key is derived using the HKDF-SHA3-256 construction, following the extract-then-expand paradigm formalized by Krawczyk (CRYPTO 2010, Springer LNCS vol. 6223, pp. 631-648) and standardized in RFC 5869. HKDF operates in two phases. The extract phase compresses potentially non-uniform input keying material into a fixed-length pseudorandom key (PRK) using HMAC as a randomness extractor: PRK = HMAC-SHA3-256(salt, IKM). The expand phase derives the output keying material from the PRK using iterated HMAC: T(1) = HMAC-SHA3-256(PRK, info || 0x01), with the output keying material taken as the first L bytes of T(1) (for a single 256-bit key, only one expansion iteration is required since SHA3-256 produces 256 bits per iteration).

The input keying material (IKM) for epoch N is constructed as the concatenation of three components. The first component is the HRoT counter attestation quote for epoch N, a signed structure produced by hrot_attest_counter() containing the current monotonic counter value and a hardware-generated signature proving the counter has been incremented exactly once since the previous epoch. The second component is the heartbeat entropy, a minimum of 256 bits of fresh min-entropy drawn from the hardware true random number generator conforming to NIST SP 800-90B (January 2018) and validated by the startup and continuous health tests specified in Section 3.3. The third component is the previous Epoch Merkle Root, defined as the root hash of the final Merkle batch committed in epoch N-1. For the genesis case (epoch 0), this value is set to H(genesis governance ceremony attestation), where the genesis governance ceremony attestation is a signed record of the initial system provisioning including the identities and cryptographic commitments of all founding governance participants.

The salt for epoch N is the epoch number encoded as a 64-bit unsigned integer in big-endian byte order. Using the epoch number as salt provides domain separation between epochs and ensures that identical IKM values (which should never occur given fresh entropy, but defense-in-depth demands consideration) would still produce distinct PRK values for different epochs.

The info field for the expand phase is constructed as the concatenation of three elements: the governance domain identifier (a fixed-length byte string uniquely identifying the TL deployment instance), the algorithm identifier (a byte string encoding the specific cryptographic suite in use, initially "SHA3-256-AES-256-GCM-v1"), and the literal ASCII string "EKR-EPOCH". The complete epoch key derivation is therefore:

```
IKM_n = hrot_attest_counter(n) || heartbeat_entropy_n || merkle_root_{n-1}
salt_n = uint64_be(n)
info  = governance_domain_id || algorithm_id || "EKR-EPOCH"
epoch_key_n = HKDF-SHA3-256(IKM_n, salt_n, info)[0:32]
```

The output is a 256-bit epoch encryption key used as the AES-256-GCM key for all EKR-encrypted fields within epoch N. **AES-256-GCM is selected as the baseline symmetric cipher** for four reinforcing reasons. First, NIST FIPS 197 (Advanced Encryption Standard, November 26, 2001; editorial update May 9, 2023) and NIST SP 800-38D (Recommendation for Block Cipher Modes of Operation: Galois/Counter Mode (GCM) and GMAC, November 2007) together define a thoroughly analyzed authenticated encryption with associated data (AEAD) construction providing both confidentiality and integrity in a single pass. Second, AES-256 achieves 128-bit security against quantum key search via Grover's algorithm, exceeding the 112-bit minimum security strength mandated by NIST SP 800-57 Part 1 Revision 5 (May 2020) for data protection beyond 2031. Third, AES-NI hardware acceleration, available on all Intel processors since Westmere (2010) and all AMD processors since Bulldozer (2011), delivers throughput exceeding 2,200 MB/s per core for AES-256-GCM (Intel "Enabling High-Performance Galois-Counter-Mode on Intel Architecture Processors" whitepaper; measured at 2,957 MB/s on Intel Gold 5412U by Calomel.org AES-NI benchmarks), enabling sub-microsecond encryption of individual Decision Log entries. Fourth, GCM mode produces a 128-bit authentication tag that detects any tampering with either the ciphertext or the associated authenticated data (the audit-accessible plaintext fields), binding the encrypted and plaintext portions of each log entry into a single authenticated structure.

The hash agility mechanism ensures that future migration from SHA3-256 to post-quantum hash functions does not break the epoch chain. The algorithm identifier encoded in the info field allows verifiers to determine which HKDF construction applies to each epoch. An epoch derived under "SHA3-256-AES-256-GCM-v1" is verified using SHA3-256 primitives; a future epoch derived under a successor algorithm identifier (for example, incorporating a post-quantum hash specified in future NIST standards) would be verified using the corresponding primitives. Since each epoch's key derivation depends only on the previous epoch's Merkle root (a hash output) and not on the previous epoch's key, the algorithm transition can occur at any epoch boundary by simply changing the algorithm identifier for the new epoch while preserving the Merkle root linkage. This provides clean algorithm migration per the guidance in NIST SP 800-57 Part 1 Revision 5, Section 5.6.4 on transitioning between algorithms, and per the draft SP 800-57 Part 1 Revision 6 (December 5, 2025) which incorporates post-quantum algorithm recommendations.

### 2.2 Subject-derived keys for commanded erasure

Per-subject encryption keys are derived using a hardened hierarchical deterministic construction that enables granular erasure at the data-subject level without destroying the epoch key. The subject key for data subject S in epoch N is computed as:

```
subject_key_n = PRF(epoch_key_n, subject_pseudonym_S || uint64_be(n))
```

where PRF denotes HMAC-SHA3-512 producing a 512-bit output, of which the leftmost 256 bits serve as the AES-256-GCM encryption key for subject S's data within epoch N. The subject pseudonym is the Hybrid Shield output for subject S, a deterministic pseudonymous identifier already committed to the Merkle tree as part of the Decision Log entry. Using the Hybrid Shield output rather than a raw subject identifier ensures that the derivation input is itself a cryptographic commitment that cannot be reversed to the data subject's identity without access to the Hybrid Shield mapping.

**Subject keys are never stored persistently.** They are re-derived on demand from the epoch key and the subject pseudonym whenever decryption of a subject's data is required. This re-derivation-on-demand model eliminates the key storage attack surface and ensures that subject keys cannot be exfiltrated from any persistent storage medium. The cost of re-derivation is one HMAC-SHA3-512 evaluation, which completes in under 1 microsecond on modern processors and introduces negligible latency compared to the subsequent AES-256-GCM decryption.

Commanded erasure for a specific data subject executes by destroying the Subject Derivation Table entry for that subject. The Subject Derivation Table (fully specified in Step 2, Section 5.2) is a mapping structure stored in TEE-shielded memory that associates each subject pseudonym with the metadata required for key re-derivation: the pseudonym value, the list of epochs in which that subject has Decision Log entries, and a status flag. When erasure is commanded, the table entry for the target subject is zeroized via hrot_zeroize(), removing the pseudonym from the derivation lookup path. Without the subject pseudonym, the PRF construction cannot be evaluated: the epoch key alone is insufficient because HMAC-SHA3-512 is a keyed function whose output is computationally indistinguishable from random without knowledge of both the key and the input. The subject's ciphertext remains in the Merkle tree as an opaque, undecryptable byte sequence; the Merkle structure is unmodified; and all other subjects' data within the same epoch remains fully recoverable because their pseudonym-to-derivation mappings are intact.

The audit read key is derived as a separate branch of the key hierarchy, ensuring that auditors can verify epoch content without obtaining the epoch encryption key or the ability to derive keys for other epochs. The audit key for epoch N is computed as:

```
audit_key_n = PRF(master_audit_secret || chain_code, uint64_be(n))
```

where master_audit_secret is a long-lived secret held by the governance authority under threshold custody (Section 2.3) and chain_code is a 256-bit domain-separation constant established during the genesis governance ceremony. The PRF output is truncated to 256 bits and serves as a decryption key for the audit-accessible encrypted layer of epoch N. Auditors receive audit_key_n for specific epochs upon authorized request. They cannot derive epoch_key_n because the audit derivation path uses a different root secret and a different derivation structure. They cannot derive audit_key_n+1 or audit_key_n-1 because the PRF is evaluated independently for each epoch number, and the one-way property of HMAC-SHA3-512 prevents inversion from an output to the corresponding input, given fixed key material. This construction provides non-correlatable per-epoch audit access: an auditor granted access to epochs 5 and 10 learns nothing about the keys for epochs 6 through 9.

### 2.3 Key lifecycle state machine with explicit TL state mapping

The lifecycle of each epoch key follows a five-state progression mapped to the TL ternary state machine. The genesis state initiates when the first epoch key is derived during system provisioning. The HRoT attestation quote for epoch 0 is generated from the initial hardware counter value, and the governance ceremony entropy (collected from multiple founding participants via verifiable secret sharing) provides the IKM in place of a prior Merkle root. During the derivation computation, the TL state register holds State 0-Transient, indicating a temporary hold condition that auto-progresses upon derivation completion. Once the epoch key is derived and installed in the TEE key cache, the state register transitions to State +1 (Execute), signaling that the DITL gate is open and the AES-256-GCM pipeline may consume the key. This State 0-Transient to State +1 transition is logged as a governance event per the No Log = No Action invariant.

The active state corresponds to normal operation under TL State +1. The epoch key is cached in TEE shielded memory and is accessible to the AES-256-GCM engine through the ternary-binary interface (Section 3.4) whenever the state register holds +1. Decision Log entries are encrypted, batched into 30-second Merkle batches, and anchored to blockchains. The epoch key remains active for the duration of the epoch, with a default duration of **24 hours** calibrated to balance forward-secrecy window size against key-derivation overhead and blockchain anchor cost.

The rotation-pending state is entered when the epoch boundary approaches (configurable threshold, default 60 seconds before boundary). The current epoch key remains at State +1, continuing to process incoming Decision Log entries, while the system begins the derivation of the next epoch key in the background. The rotation-pending state ensures a seamless handoff: the incoming epoch key is derived and installed in a secondary TEE key cache slot before the current epoch closes. The state transition from rotation-pending to the next epoch's active state requires explicit State +1 validation: the new epoch key is released to the AES-256-GCM engine only after the DITL gate confirms State +1 for the new epoch, and the previous epoch key is immediately zeroized upon confirmation.

The escrowed state applies to epoch keys held under a threshold custody scheme for disaster recovery. Escrowed keys are split using Shamir's Secret Sharing with a threshold of k-of-n (configurable, default 3-of-5) and distributed to governance custodians. Each share is stored in the custodian's personal HRoT (HSM or hardware token). An escrowed key carries TL State -1 for direct operational use, meaning it cannot be loaded into the AES-256-GCM pipeline without first being reconstituted through the threshold scheme and validated through a governance ceremony that transitions the reconstituted key to State +1. This prevents a single custodian from unilaterally activating an escrowed key, per the NIST SP 800-57 Part 1 Revision 5 recommendation that archived keys require multi-party authorization for recovery (Section 8.3.1).

The zeroized state is the terminal state, corresponding to TL State -1 permanent. Once an epoch key is zeroized, the state register for that epoch is permanently set to -1, the key material is irrecoverably destroyed via hrot_zeroize(), and the DITL gate for that epoch is permanently closed. No state transition from zeroized is possible. The Destruction Event logged before zeroization serves as the permanent audit record that the key existed, was active for a defined period, and was destroyed under specific governance authorization.

Forced rotation triggers override the default 24-hour epoch duration and initiate immediate epoch boundary closure and key rotation. Three conditions mandate forced rotation. First, heartbeat entropy failure: if the continuous health test (Section 3.3) detects entropy source degradation, the current epoch closes immediately because subsequent key derivations cannot be trusted to incorporate sufficient randomness. Second, tamper detection: if the HRoT's tamper mesh (Section 3.1) reports a physical intrusion event, the current epoch key is zeroized without rotation (State -1 immediate), and a new epoch cannot begin until the tamper condition is resolved through governance intervention (State 0-Suspended). Third, governance command: an authorized governance participant may command immediate epoch rotation through the governance pipeline, producing a signed rotation order that is logged as a governance event before execution. High-velocity operational contexts (high-frequency trading infrastructure, real-time autonomous decision systems) may configure sub-hour epoch durations, but every epoch must contain at least one closed Merkle batch anchored to at least one blockchain to maintain the evidentiary chain. An epoch containing zero committed batches is architecturally invalid and will trigger State 0-Suspended on the subsequent epoch's derivation attempt.

## Section 3: Hardware root of trust and timing architecture

### 3.1 HRoT epoch counter and physical security

The epoch counter is the foundational integrity primitive of the key hierarchy. It must be monotonic (each increment produces a strictly greater value), non-resettable without multi-party governance authorization, and hardware-attested (each counter read produces a signed quote proving the counter value was generated by authentic hardware in a non-tampered state). The HRoT interface abstracts this counter behind three functions: hrot_get_counter() returns the current counter value, hrot_attest_counter() returns the counter value with a hardware-generated attestation signature, and hrot_zeroize() triggers immediate destruction of all key material held by the HRoT.

The baseline TPM 2.0 binding uses the NV Index facility defined in the Trusted Computing Group TPM 2.0 Library Specification (most recent: Version 185, March 12, 2026; international standard ISO/IEC 11889:2015). An NV Index is defined with the TPMA_NV_COUNTER attribute, which designates the index as a 64-bit monotonic counter. The **TPM2_NV_Increment command** (Part 3, Section 31.8, Tables 215-216) increments the counter by exactly one. The TPM guarantees that when a counter-type NV index is first incremented, its value is initialized to one greater than the largest value any counter has ever held on that TPM, ensuring monotonicity even if the NV index is deleted and recreated. The TPM2_NV_Certify command (Part 3, Section 31.16, Tables 231-232) produces an attestation signature over the NV index data, returning a TPMS_NV_CERTIFY_INFO structure (Part 2, Table 119) signed by an attestation key resident in the TPM. This signature constitutes the hardware attestation quote used as the first IKM component in epoch key derivation.

Counter recovery after power loss is handled by the TPM's NV storage durability model. Standard (non-orderly) NV counters write to non-volatile storage on every increment, ensuring the counter value survives unexpected power loss at the cost of NV wear. For deployments with high-frequency counter increments, the TPMA_NV_ORDERLY attribute enables hybrid operation: the counter value is maintained in volatile memory with periodic NV commits and a guaranteed NV write on orderly shutdown via TPM2_Shutdown. In a TL deployment, since the epoch counter increments only once per epoch (default every 24 hours), NV endurance is not a practical concern, and the standard non-orderly mode with per-increment NV writes provides the strongest durability guarantee. The startup attestation sequence requires that upon power restoration, the system execute hrot_attest_counter() and verify the returned counter value equals the expected next epoch number before accepting any Decision Log entries, entering State 0-Suspended if the values diverge.

The NV Index authorization policy prevents unauthorized counter manipulation. The counter's NV index is defined with a policy requiring TPM2_PolicyCommandCode restricting operations to TPM2_NV_Increment and TPM2_NV_Certify, combined with TPM2_PolicySecret binding the authorization to a governance credential that is itself protected by the TPM's key hierarchy. The TPMA_NV_POLICY_DELETE attribute is set, ensuring that TPM2_NV_UndefineSpace (which could delete the counter) fails; only TPM2_NV_UndefineSpaceSpecial with platform authorization can remove the index, and platform authorization is bound to physical presence or governance multi-party ceremony.

The ROCA vulnerability (CVE-2017-15361), documented by Nemec, Sys, Svenda, Klinec, and Matyas in "The Return of Coppersmith's Attack: Practical Factorization of Widely Used RSA Moduli" (24th ACM Conference on Computer and Communications Security, pp. 1631-1648, 2017, DOI 10.1145/3133956.3133969), demonstrated that Infineon's proprietary RSA key generation library produced RSA moduli with exploitable algebraic structure, enabling practical factorization of 2048-bit keys at modest computational cost. **Approximately 25% of TPMs globally were affected**, including devices with FIPS 140-2 and Common Criteria certifications. The TL specification addresses the ROCA vulnerability class through three requirements. First, all asymmetric keys generated within the HRoT must use standardized probabilistic prime generation with Miller-Rabin primality testing as specified in NIST SP 800-56B Revision 2, not proprietary fast-prime optimizations. Second, the TL deployment validation procedure includes ROCA fingerprint detection (the vulnerability is detectable in microseconds via discrete logarithm computation over the small composite modulus used by the affected library) for all RSA keys generated by the TPM, with any positive detection triggering immediate key regeneration and State 0-Suspended. Third, the post-quantum migration path replaces RSA-based attestation with ML-DSA (FIPS 204) signatures, eliminating dependence on the integer factorization hardness assumption entirely.

Physical tamper protection is required at all security levels. The HRoT module must include a tamper mesh: a conductive trace grid overlaying the protected circuitry that continuously measures electrical properties (conductivity, impedance, propagation delay) and compares them against factory-calibrated reference values. Any physical manipulation (drilling, probing, desoldering, micro-milling) that disrupts the mesh triggers immediate hrot_zeroize(), destroying all epoch keys, attestation keys, and counter authorization credentials. This requirement aligns with FIPS 140-3 Security Level 3, which mandates tamper detection and response mechanisms that zeroize all plaintext Critical Security Parameters upon detection. The Thales Luna 7 HSM implements this through a multi-chip standalone module enclosed in a tamper-evident, tamper-responsive enclosure validated at FIPS 140-3 Level 3 (NIST Certificate 4962, April 2024). The Entrust nShield HSM series provides tamper detection that continues operating for at least three years on battery backup without external power, logging the date and time of tamper events. Fault injection mitigation (voltage glitching, clock glitching, electromagnetic fault injection) is addressed through internal voltage regulation, redundant computation of critical security operations, and tunable replica circuits for dynamic fault detection, following the countermeasure taxonomy in the Trusted Firmware-M Fault Injection Mitigation guidance.

### 3.2 Timing budget and the Fast Lane Prohibition

The Fast Lane Prohibition is a named mandatory constraint of the TL architecture. It states: **direct HRoT non-volatile storage access is prohibited in the Fast Lane decision path** (sub-2ms latency budget for per-decision cryptographic operations). This prohibition exists because TPM 2.0 NV operations and attestation signing carry latencies measured in tens to hundreds of milliseconds, which would violate the 2ms Fast Lane budget by one to two orders of magnitude. Benchmarks by Kristoffersen (stiankri.substack.com/p/tpm-performance, April 2024, testing multiple Intel and discrete TPM platforms) measured ECDSA P256 signing at 8-9 operations per second on Intel i5-1135G7 firmware TPM (approximately 111-125ms per operation) and 2-7 operations per second on discrete TPMs (approximately 143-500ms per operation). The wolfTPM benchmark suite (wolfSSL wolfTPM, github.com/wolfSSL/wolfTPM) measured ECDSA P256 signing at 35-72ms per operation and HMAC-SHA256 at 86-138 KB/s throughput across multiple TPM chips. Chrome browser measurements for Device Bound Session Credentials (WICG/DBSC specification) found TPM ECDSA P256 latency of 200ms at the 50th percentile and 600ms at the 95th percentile.

The Fast Lane path therefore uses TEE-cached epoch keys. The current epoch key, once derived and validated at State +1 during epoch initialization, is installed in TEE shielded memory (Intel SGX Enclave Page Cache, ARM TrustZone secure memory, or RISC-V Keystone PMP-protected memory). All per-decision encryption operations read the key from this cache rather than from the HRoT. HRoT attestation occurs asynchronously in the Governance Lane, once per 30-second Merkle batch close, where the 5-15ms TPM latency (or sub-millisecond HSM/TEE latency) is absorbed without impacting decision throughput. Any violation of the Fast Lane Prohibition (a code path that attempts direct HRoT NV access during a Fast Lane operation) triggers immediate State 0-Suspended and generates a governance alert. The violation itself is logged as a security event; the system enters Epistemic Hold pending governance review of the violating code path.

The complete timing budget for Fast Lane operations is presented below. All Fast Lane components must complete within the 2ms total budget. Governance Lane operations execute asynchronously and do not contribute to per-decision latency.

| Operation | Latency | Budget Share | Execution Context | Source |
|---|---|---|---|---|

This margin accommodates scheduling jitter, cache misses, context switch overhead, and defensive timing padding against side-channel analysis.

The Thales Luna 7 HSM alternative achieves sub-millisecond attestation because HSM symmetric and asymmetric operations execute on dedicated cryptographic accelerator hardware within a FIPS 140-3 Level 3 validated boundary, delivering **10,000 RSA-2048 signatures per second and 22,000 ECC P256 signatures per second** on the A790 model (Thales Luna Network HSM 7 product brief). The RISC-V Keystone TEE alternative (Lee, Kohlbrenner, Shinde, Asanovic, and Song, "Keystone: An Open Framework for Architecting Trusted Execution Environments," EuroSys 2020, DOI 10.1145/3342195.3387532) achieves less than 1% computation overhead for in-enclave operations through Physical Memory Protection (PMP) register-based isolation without CPU modifications, with the Security Monitor comprising only 1,600 lines of code. Keystone's enclave entry/exit cost is dominated by PMP reconfiguration and TLB flush, but cache-resident key reads within an active enclave execute at native DRAM speed with no per-access overhead.

### 3.3 Heartbeat entropy source and health testing

The heartbeat entropy source provides the non-recoverable randomness that guarantees epoch key independence and, by extension, forward secrecy. Without fresh, validated entropy at each epoch derivation, two epochs could produce correlated keys, violating the forward-secrecy guarantee. The entropy source conforms to NIST SP 800-90B (Recommendation for the Entropy Sources Used for Random Bit Generation, January 2018, authored by Turan, Barker, Kelsey, McKay, Baish, and Boyle), which specifies requirements for noise sources, conditioning components, and health testing.

The minimum entropy requirement is **256 bits of min-entropy per epoch key derivation**. Min-entropy, the conservative entropy measure used in SP 800-90B, represents the negative logarithm (base 2) of the probability of the most likely output, ensuring security even against adversaries who can predict the most probable output values. The 256-bit threshold matches the AES-256 key length, ensuring the derived key's effective strength is not bottlenecked by entropy input.

Startup testing requires that before the first epoch key derivation after any power cycle or system reset, the hardware TRNG must generate and pass health tests on a pool of at least 4,096 noise source samples per SP 800-90B Section 4.3 startup testing requirements. The TL architecture imposes an additional requirement beyond SP 800-90B: a validated block of 256 bits of conditioned entropy must be produced and verified before the first epoch key derivation is permitted. During startup testing, the TL state machine holds State 0-Transient, auto-progressing to State +1 only upon successful entropy validation. If startup testing fails (health test detects bias, stuck-at, or correlation in the noise source), the system enters State 0-Suspended and cannot derive any epoch key until the entropy source is replaced or repaired under governance authorization.

Continuous health testing monitors the entropy source during ongoing operation. The TL architecture mandates a heartbeat entropy test of 64 bits every 100 milliseconds, where 64 bits of raw noise source output are evaluated by the SP 800-90B continuous health tests (Repetition Count Test and Adaptive Proportion Test, Section 4.4). The Repetition Count Test detects a single output value appearing more than a threshold number of consecutive times, calibrated to the noise source's assessed entropy rate. The Adaptive Proportion Test detects a single value appearing with disproportionate frequency over a sliding window. Both tests are applied to raw noise source output before any conditioning, ensuring they detect degradation at the physical noise generation layer. If either test fails, the system transitions to State 0-Suspended and initiates automatic emergency epoch rotation: the current epoch closes immediately, the current epoch key is zeroized, and no new epoch key derivation proceeds until a fresh startup test passes on a validated entropy source. This emergency rotation occurs before the next log entry is accepted, ensuring no Decision Log entry is ever encrypted with a key derived from suspect entropy.

The TRNG is implemented on a **separate power domain** from the log generation pipeline. This physical separation ensures that a power fault, voltage glitch, or electromagnetic interference event targeting the log pipeline cannot simultaneously corrupt the entropy source, and that a compromised log pipeline cannot manipulate the TRNG's power supply to bias its output. The TRNG output path has no network-accessible interface: entropy flows exclusively through a hardware bus from the TRNG to the TEE's entropy conditioning component, with no software-addressable register on any network-facing interface that could read or inject entropy values. This eliminates the heartbeat injection attack vector in which a network adversary substitutes controlled values for true random entropy. Side-channel mitigation against electromagnetic emanation analysis of the TRNG output path is provided by the same tamper mesh that protects the HRoT (Section 3.1), which detects the probe placement required for close-range electromagnetic analysis.

### 3.4 Ternary-binary interface governance semantics

The ternary-binary interface is the hardware-software boundary where TL's three-valued governance state machine controls access to binary cryptographic accelerators. This interface enforces the invariant that **the epoch key must not be readable by the AES-256-GCM engine unless the ternary state register holds State +1** (Execute). The state register is a three-valued hardware register (physically represented as a two-bit encoding: 00 for State 0-Transient, 01 for State 0-Suspended, 10 for State +1, 11 for State -1) that gates the key bus connecting the TEE key cache to the AES-256-GCM pipeline.

When the state register holds State +1, the key bus gate is physically open (conducting), and the AES engine can read the epoch key from the TEE cache to initialize its key schedule. The DITL gate (Decision Integrity Ternary Logic gate) verifies State +1 before each key schedule initialization, ensuring that a state transition to 0 or -1 that occurs between two consecutive encryption operations is respected even if the AES pipeline has capacity for another operation.

When the state register holds State 0 (either Transient or Suspended), the key bus gate is physically non-conducting. The epoch key is not merely logically denied through a software access-control check; the bus impedance is driven to a state that makes the key electrically unreadable at the AES engine's input pins. This physical gating prevents hardware-level bypass attacks in which a compromised AES engine firmware or a fault injection into the access-control logic could circumvent a software-only denial. The distinction between State 0-Transient and State 0-Suspended does not affect the gate behavior (both gate the key bus identically) but affects the governance response: State 0-Transient auto-resolves when the pending operation (key derivation, Merkle batch close) completes, while State 0-Suspended requires explicit governance intervention to resolve.

When the state register transitions to State -1 (Block), the key bus undergoes immediate zeroization. The gate not only becomes non-conducting but actively drives the key bus lines to ground, clearing any residual charge that might represent key bits in transit. If an AES-256-GCM encryption operation is in progress when State -1 is asserted, the key bus zeroization interrupts the AES pipeline regardless of pipeline state. The partially encrypted output is discarded (not committed to the ledger), and the AES engine's internal key schedule registers are flushed. The -1 transition's effect on the key bus must take effect **within one clock cycle** of the state register update, regardless of AES pipeline depth. This one-cycle guarantee requires that the state register drives the key bus gate through combinational logic (no intervening flip-flop stages), ensuring the gate responds within the same clock edge that updates the state register.

The transition from State 0 to State +1 has the opposite timing constraint: the key bus gate must fully open and the key must be stable at the AES engine's input before the AES key schedule computation begins. This is enforced by requiring a minimum hold time of two clock cycles between the State +1 assertion and the first AES key schedule round, implemented as a pipeline stall in the AES engine that checks the state register on each cycle and proceeds only after observing two consecutive cycles of State +1. This two-cycle hold prevents a transient state-register glitch from allowing a single-cycle key read during an otherwise-gated state.

The physical implementation details of the gate (transistor-level design, voltage thresholds, impedance specifications, electromagnetic shielding of the key bus traces) are deferred to the Hardware Architecture folder, which specifies the gate design for each HRoT platform variant (TPM 2.0, Thales Luna 7, RISC-V Keystone). The specification in this section defines the functional and timing requirements that any physical implementation must satisfy. Formal verification of the gate logic against these timing requirements using model checking (e.g., TLA+ or UPPAAL timed automata) is a validation prerequisite before any TL deployment clears the governance ceremony for production activation.
